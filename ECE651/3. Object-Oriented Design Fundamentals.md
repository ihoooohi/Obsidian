# 面向对象基础复习

## Class 的使用

讲义的第一句话非常重要：

> A class describes a type of object we might create.

### Class 是什么？

- 描述一种 " 对象类型 "
- 包含：
	- **fields（状态）**
	- **methods（行为）**
- 用 `new` 创建对象

### Class 的三个核心组成

#### Fields——状态

讲义例子：

> A Point has x and y
> A Ship has name, location, crew

例如：

```java
class Point {  
    private int x;  
    private int y;  
}
```

👉 fields = 对象 " 拥有什么 "

#### Methods——行为

讲义说：

> Methods are verbs

```java
class Point {  
    void move(int dx, int dy) { … }  
    double computeDistance(Point other) { … }  
}
```

👉 methods = 对象 " 能做什么 "

#### Constructor——创建对象

```java
Point p = new Point(3, 5);
```

👉 new 的本质：分配内存 + 初始化字段

> [!NOTE] 关键理解（ECE651 视角）
> 在软件工程里：
>
> > class 是抽象建模工具
>
> 它不是 " 语法结构 "，
> 而是对现实世界或问题域的一种建模。
>
> 例如 Battleship：
>
> - Placement 是一个概念
> 
> - Coordinate 是一个概念
> 
> - Ship 是一个概念
> 
> 
> 这不是编程问题，是建模问题。

## Has-A vs Is-A（Page 6–10）

这是面向对象最重要的设计判断之一。

### Has-A（组合）

讲义说：

> Has-A relationships are implemented as fields

例子：

```java
class Zoo {  
    private List<Animal> animals;  
}
```

Zoo has animals 👉 用字段实现。

#### 特征

- 部分 - 整体关系
- 组合关系
- 更灵活
- 不暴露内部

#### 在 Battleship 中

```cpp
class Board {  
    private List<Ship> ships;  
}
```

Board has ships.

✔ 正确

✔ 易扩展

✔ 易替换内部结构

### Is-A（继承）

讲义例子：

- class AircraftCarrier extends Ship
- AircraftCarrier is-a Ship.

#### 特征

- 子类是父类的 " 更具体类型 "
- 接口必须是父类的子集
- 必须满足 LSP

> [!question] 如何判断？
>
> 讲义给了三个判断标准（Page 8）：
>
> ##### 1. 是否 " 更具体类型 "？
>
> - SportsCar is-a Car ✔
> 
> - Person is-a Head ❌
> 
> ##### 2. 子类是否能替代父类？
>
> 这是 Liskov Substitution Principle (LSP)：
>
> > 如果 S 是 T 的子类
> > 那么 S 应该可以替代 T 使用
>
> 如果不能替代 → 继承是错的。
>
> ##### 3. 工程现实建议
>
> 在现代设计里：
>
> > 优先使用 Has-A
> > 谨慎使用 Is-A
>
> 因为：
>
> - 继承 = 强耦合
> 
> - 父类改动 → 子类可能全炸
> 
> - 测试困难
> 
> 组合更安全。

## Encapsulation（封装）（Page 11–16）

讲义这一部分是软件工程的核心思想。

### 什么是封装？

讲义说：

> combining data with the methods that act on them

也就是：

- 数据和操作数据的方法放在一起
- 不允许外部直接操作数据

### 新手错误：滥用 getter / setter

讲义 Zoo 例子：

```java
class Zoo {  
    private List<Animal> animals;  
    public List<Animal> getAnimals() { … }   // ❌  
}
```

为什么错？因为外部代码现在知道：

- Zoo 用的是 List
- Zoo 的内部结构

这违反封装。

### 正确方式

讲义建议：

```java
addAnimal()  
removeAnimal()  
getAnimalIterator()
```

外部只知道：

- Zoo 可以 " 添加动物 "
- Zoo 可以 " 遍历动物 "

但不知道：

- 是 List 还是 HashMap

> [!NOTE] 关键思想
> 封装的真正目的不是 "private"，而是：
>
> > 隐藏实现细节
> > 保证将来可修改性
>
> 讲义 Page 16 提到，以后如果改成：
>
> ```java
> HashMap<Integer, Animal>
> ```
>
> 外部代码不需要改。这叫：
>
> > Implementation hiding

## Static 的使用（Page 17–20）

讲义标题非常直白：

> STATIC (USUALLY BAD)

### static 是什么？

讲义解释：

- static field = 属于类
- static method = 不依赖具体对象

例如：

```java
class Math {  
    static int max(int a, int b) { … }  
}
```

### 为什么通常是不好的？

因为它：

- 破坏面向对象思想
- 不属于某个实例
- 很难测试
- 很难 mock
- 常常变成 " 全局变量 "

在软件工程中：

> static = 全局状态

全局状态 = 高耦合 + 难维护

> [!question] 什么时候可以用？
>
> 讲义列出 4 种合理情况：
>
>
>
> ##### 1. 纯函数
>
> 例如：
>
> ```java
> Math.sqrt()  
> Math.max()
> ```
>
>
> 特征：
>
> - 无状态
> 
> - 只依赖输入
> 
> - 无副作用
> 
> 
> 
> 
> ##### 2. 生成全局 ID
>
> 例如：
>
> ```java
> private static int nextId;
> ```
>
> 用于生成唯一编号。
>
>
>
> ##### 3. super() 参数计算
>
> 因为 Java 要求：
>
> ```
> super(…)
> ```
>
> 必须是构造函数第一行。可以调用 static 方法计算参数。
>
>
>
> ##### 4. Singleton（以后讲）
>
> 但讲义强调：
>
> > 要很谨慎的使用。

# 单一职责 + 低耦合高内聚原则的应用

## 什么是 Single Responsibility？

最经典表述：

> A class should have only one reason to change.

一个类只应该有一个 " 变化的原因 "。

### 关键理解

" 一个职责 " 不是 " 只做一件小事 "。而是：

> 所有功能围绕同一类变化。

### 举例

错误例子：

```java
class OrderProcessor {  
    void readInput()  
    void processOrder()  
    void printReceipt()  
}
```

问题：

- 输入变化 → 要改
- 业务规则变化 → 要改
- 输出格式变化 → 要改

一个类有三个变化来源 ❌

## Low Coupling + High Cohesion

这两个原则总是一起出现。

### Cohesion（内聚）

内聚高 = 类内部的方法和数据强相关。

低内聚 =

- 类像 " 工具箱 "
- 东一块西一块

### Coupling（耦合）

耦合高 = 一个模块依赖另一个模块的内部细节。

低耦合 =

- 只依赖接口
- 不依赖实现

## 讲义中的 C 语言例子（Page 22–30）

讲义用一个 C 程序举例，非常经典。

```c
array_t *getSortedStrings(File *f){
	array_t *arrayToSort = malloc(sizeof(*arrayToSort));
	arrayToSort->array = NULL;
	arrayToSort->n = 0;
	char *curr = NULL;
	size_t sz = 0;
	size_t i = 0;
	while(getline(&curr, &sz, f) >= 0){
		arrayToSort->array = realloc(arrayToSort->array, (i + 1) * sizeof(*arrayToSort->array));
		arrayToSort->array[i] = curr;
		curr = NULL;
		i++;
	}
	
	arrayToSort->n = i;
	sortData(arrayToSort->array, arrayToSort->n);
	return arrayToSort;
}

  

void printSorted(FILE *f){
	array_t *sorted = getSortedStrings(f);
	for(size_t i = 0; i < sorted->n; i++){
		printf("%s", sorted->array[i]);
		free(sorted->array[i]);
	}
	free(sorted->array);
	free(sorted);
}
```

原始代码做 4 件事：

1. 读取文件
2. 排序
3. 打印
4. 释放内存

**原始设计的问题**

讲义 Page 23–25 分析：

- 1 + 2 强耦合
- 3 + 4 强耦合
- 3 不能单独使用
- 4 不能单独使用

你只能：

```
(1+2)  
或  
(1+2+3+4)
```

不能只打印 不能只排序

### 为什么这是坏设计？

因为：

> 功能不能独立复用。

### 改进版本（Page 26–27）

讲义将代码拆分成：

```c
readStringsFromFile()  
sortData()  
printArray()  
freeArray()
```

然后：

```c
printSorted() {  
    read  
    sort  
    print  
    free  
}
```

**这就是解耦的本质。** 讲义 Page 27 强调：

> We can still combine behaviors.
> The important part is we can do them separately.

组合不是问题。问题是：

> 不能拆开。

## Burger Combo 类比（Page 28–30）

讲义用了一个非常好的比喻。

> 假如有这样一个餐厅，你只能买套餐：Fries + Burger + Shake
> 但你只想要 shake。

问题不在于套餐存在。问题在于：不能单点。

### 对应到代码

坏设计：

```c
Combo 1 = Read + Sort  
Combo 2 = Read + Sort + Print + Free
```

不能单点：

- 不能只排序
- 不能只打印
- 不能只 free

### Page 32 的关键总结

讲义给出一条极重要的工程规则：

> [!important] Always separate:
> 1. Input
> 2. Operation
> 3. Output
> 4. Cleanup

#### 为什么？

因为这四类变化来源完全不同。

|类型|变化来源|
|---|---|
|Input|文件格式变化|
|Operation|业务逻辑变化|
|Output|UI 变化|
|Cleanup|资源管理策略|

> [!NOTE] 工程本质理解
> Single Responsibility 的真正目标是：
>
> > 把 " 变化的方向 " 分离。
>
> Coupling 的真正问题是：
>
> > 一个变化会连锁触发其他变化。

### 映射到 Battleship 项目

我们来做一次设计体检。

#### 错误示例

```java
void playTurn() {  
    System.out.println("Enter coordinate:");  
    String input = scanner.nextLine();  
    Placement p = new Placement(input);  
    board.placeShip(p);  
    System.out.println(board);

```

这个函数包含：

1. 输入读取
2. 输入解析
3. 业务逻辑
4. 输出展示

四种职责。

```java

void displayBoard(OutputStream out)

然后：

void playTurn() {  
    Placement p = readPlacement(in);  
    placeShip(p);  
    displayBoard(out);  
}
```

# 输入格式解耦

## Drive-Thru 场景（JSON 输入）

我们来设计一个自动点单系统。顾客按点单按钮，系统将订单以 JSON 格式发送到店内后台的自动处理订单系统。讲义 Page 33 给出 JSON：

```json
{  
  "items": [  
    { "type": "burger", "count": 2 },  
    { "type": "shake", "variety": "chocolate", "count": 1 }  
  ],  
  "payment": { "type": "cash", "amount": 7.93 }  
}
```

代码写成：

```java
JsonObject order = JsonReader.readObject(orderStr);  
JsonArray items = order.getJsonArray("items");  
JsonObject food = items.getJsonObject(i);  
food.getString("type");  
food.getInt("count");
```

### 新手写法（Page 39）

```java
FoodBag packOrder(String orderString) {  
    JsonObject order = JsonReader.readObject(orderStr);  
    JsonArray items = order.getJsonArray("items");  
    …  
    fb.addFood(food.getString("type"), variety);  
}
```

讲义明确说：

> 处理与输入格式过于紧密耦合。

![[Pasted image 20260303131822.png|500]]

### 什么是 " 输入格式耦合 "？

输入格式耦合 = 业务逻辑直接操作 JSONObjects / XMLElements。

换句话说：

- 业务逻辑知道字段名
- 业务逻辑知道 JSON 结构
- 业务逻辑依赖输入表示形式

### 为什么这是严重问题？

讲义 Page 41 引入新情况。如果网站改用 XML：

```xml
<order>  
  <items>  
    <food type="burger" count="2"/>  
    …  
  </items>  
</order>
```

![[Pasted image 20260303131838.png|471]]

问题来了：

- XML 不是 JsonObject
- 结构不同
- API 不同

### 如果你紧耦合 JSON 会发生什么？

你只能：

1. 写大量转换代码
2. 或者把 XML 转 JSON 再处理
3. 或重写所有业务逻辑

这叫：

> 表示变化导致逻辑变化

严重违反：

- Single Responsibility
- Low Coupling

### 讲义给出的正确设计（Page 45）

讲义核心句：

> Decouple program state from input representation.

**正确结构应该是：**

![[Pasted image 20260303131920.png|449]]

```
Input(JSON)   →   Order Object  
Input(XML)    →   Order Object
```

而不是：

```
Input(JSON) → Processing(JSONObject)
```

### 正确架构图

```
JSON Parser → Order  
XML Parser  → Order  
  
Order → Processing
```

Processing 不再知道：

- JSON
- XML
- 字段名
- 解析方式

### 为什么要解耦输入格式？

给你四个工程级理由：

#### 1️. 输入格式会变化

- JSON
- XML
- CSV
- Binary
- REST
- GraphQL

业务逻辑不应该每次跟着重写。

#### 2. 输入只是 " 表示 "

JSON 只是：

> 一种外部表达形式

而程序内部应该有：

> 稳定的领域模型（Domain Model）

#### 3️. 类型安全

JSONObject 只有：

```java
getString()  
getInt()
```

没有类型语义。

而：

```java
class Order  
class Food  
class Payment
```

有：

- 强类型
- 行为
- 不变量

#### 4️. 可测试性

- 如果业务逻辑依赖 JSON：测试必须构造 JSON 字符串。
- 如果业务逻辑依赖 Order：测试可以直接：

```java
new Order(…)
```

更干净。

## 如何实现输入解耦？

这是最关键的问题。

### Step 1：定义领域模型（Domain Model）

```java
class Order {  
    private List<Food> foods;  
    private Payment payment;  
}  
  
class Food {  
    private String type;  
    private int count;  
    private String variety;  
}
```

这是你的 " 内部状态 "。

### Step 2：写专门的输入解析器

#### JSON 版本

```java
class JsonOrderParser {  
    Order parse(String jsonStr) {  
        JsonObject obj = …  
        return new Order(…);  
    }  
}
```

#### XML 版本

```java
class XmlOrderParser {  
    Order parse(String xmlStr) {  
        XMLElement element = …  
        return new Order(…);  
    }  
}
```

### Step 3：业务逻辑只依赖 Order

```java
class OrderProcessor {  
    FoodBag process(Order order) {  
        …  
    }  
}
```

OrderProcessor 不知道：

- JSON
- XML
- 输入来源

### 抽象级别再提升：接口解耦

可以进一步：

```java
interface OrderParser {  
    Order parse(String input);  
}
```

然后：

```java
OrderParser parser = new JsonOrderParser();  
Order order = parser.parse(input);  
processor.process(order);
```

这叫：

> 输入解耦 + 多态

# MVC 模式

我们先回顾前一页（Page 45）：

讲义说：

> 将程序状态与输入格式解耦。

它给出结构：

```
JSON → Order  
XML  → Order
```

这时候问题来了：

> 既然我们已经有了独立的 Order / Food / Payment
> 那么整个程序应该如何组织？

这正是 Page 46 开始出现：

> MODEL VIEW CONTROLLER (MVC)

换句话说：输入解耦只是第一步。下一步是：

> 整个系统如何分层？

## 什么是 MVC？

我们的逻辑是：

1. 不要把业务逻辑绑在 JSON 上。
2. 要有内部领域模型。
3. 那么：
	
	- 谁负责输入？
	- 谁负责业务？
	- 谁负责展示？

答案：

- Controller → 输入
- Model → 业务
- View → 展示

输入解耦是 MVC 的第一步。

Page 46 说：

> MVC: Model View Controller
> 这是一种设计范式，它认为应该将模型、视图和控制器分开。

它是一种**架构模式**（Architectural Pattern）。不是语法，不是类设计，而是系统组织方式。

## MVC 三个部分是什么？

讲义 Page 47–49 分别介绍三部分。

### Model（模型）

Page 47 定义：

> Model: Data and business logic

![[Pasted image 20260303132233.png|633]]

Model 包含：

- 程序状态
- 业务规则
- 核心逻辑

例如：

```java
Processing order
Processing payment
Pack food into a foodbag
```

在 Battleship 中：

- Board
- Ship
- Placement
- Coordinate
- GameState

### View（视图）

Page 48 定义：

> View: How we display it

![[Pasted image 20260303132315.png|630]]

职责：

- 展示 Model 状态
- 不改变 Model
- 不做业务逻辑

例如：

- 打印棋盘
- GUI 渲染
- 输出文本

### Controller（控制器）

Page 49 定义：

> What initiates changes in the model's state

![[Pasted image 20260303132337.png|644]]

职责：

- 接收输入
- 解析输入
- 调用 Model 改变状态

例如：

```java
playTurn()
```

Controller 不直接修改 UI。

Controller 不包含业务规则。

它只是：

> 把输入转换成对 Model 的调用。

## MVC 的核心：围绕 " 状态变化 "

Page 50 给出一个关键图：

![[Pasted image 20260227215936.png|500]]

讲义强调一句话：

> Remember that it is all about change.

MVC 关注的不是数据结构，而是：

> 谁负责触发状态变化？

## 为什么要 MVC？

我们从工程角度分析。

1. **输入形式会变化**

- CLI
- GUI
- Web
- Network

Controller 可以变，Model 不变。

2. **输出形式会变化**

- 文本
- 图形
- API

View 可以变，Model 不变。

3. **业务规则变化**

- 只改 Model，不动 View。

> [!NOTE] 讲义特别提醒（Page 50 / 51）
> ##### 提醒 1
>
> 讲义说：
>
> > Many novices:
> >
> > 1. miss the point of MVC
> > 
> > 2. apply it badly: code in wrong part, tight coupling between them
>
> 这是非常真实的。
>
> 常见错误：
>
> - 在 View 里写逻辑
> 
> - Controller 直接修改 View
> 
> - Model 打印输出
> 
> 这些都违反 MVC。
>
> ##### 提醒 2
>
> 讲义说：
>
> > View and Controller often go together.
>
> 例如：
>
> - Text view + text input
> 
> - GUI view + GUI input
> 
> 
> 但：
>
> > 逻辑上仍然要分离。
>
> 物理上可能在一个类里，但职责必须清晰。

## 用 Battleship 举例（非常重要）

假设我们做 CLI 版本。

### 错误设计 1

```java
class Board {  
    void placeShip(String s) {  
        System.out.println("Placing ship…");  
        …  
    }  
}
```

Board 在打印东西。Board 现在知道 UI，违反 MVC。

### 错误设计 2

```java
class BoardTextView {

        if (ship overlaps) {  
            throw error;  
        }  
    }  
}
```

View 在写业务逻辑。违反 MVC。

### 正确设计

Model：

```java
class Board {  
    void placeShip(Placement p)  
}
```

Controller：

```java
Placement p = readPlacement(input);  
board.placeShip(p);
```

View：

```java
display(board);
```

### 字符串与对象 (Page 52–57)

讲义特别强调：

> 字符串是危险的数据结构

例如：

```
"A3H"
```

不应该一直保存字符串，应该立即解析成：

```
Placement  
  - Coordinate  
  - Orientation
```

原因：

- 类型更安全
- 更易扩展
- 不重复解析
- 代码更清晰

# 异常与错误处理解耦

## 551 的错误处理方式（Page 63）

讲义给出典型写法：

```java
Placement(String descr) {  
    if (some error) {  
        System.err.println("The placement was invalid");  
        System.exit(1);  
    }  
}
```

这在 551 是可以接受的。但在软件工程里是严重问题。

## 这种写法有什么问题？

### 1. 强耦合

构造函数：

- 决定错误信息
- 决定错误处理方式
- 决定程序终止

也就是说：

> 业务代码 = UI + 控制流

违反 SRP。

### 2️. 无法复用

如果以后：

- GUI 版本
- 网络版本
- 批量测试版本

你不希望一出错就退出程序。但 `System.exit()` 已经写死。

### 3️. 无法测试

如果构造函数里：`System.exit(1);` 那测试：`new Placement("A0");`

会直接终止 JVM。你甚至无法写：

```
assertThrows(…)
```

### 4️. 决策层级错误

讲义 Page 81 说：

> We need to make that decision **as high up** as possible.

退出程序应该在：

- `main()`
- 最外层

而不是在底层库代码里。

## 为什么要解耦异常与错误处理？

核心原因：

> 错误产生的地方
> 不应该决定如何处理错误。

不同场景下：

|场景|处理方式|
|---|---|
|CLI 游戏|打印 + 重试|
|GUI|弹窗|
|Web API|返回 400|
|单元测试|assertThrows|
|main|exit|

如果构造函数里 exit：所有场景都被锁死。

## Java 的解决方案：Exception（Page 59–62）

讲义说：

> 异常将 **错误处理** 与 **产生错误的代码** 分离开来。

### 异常机制如何实现解耦？

当发生错误：

```java
throw new IllegalArgumentException("Invalid placement");
```

这里发生了什么？

1. 当前函数停止
2. 异常沿调用栈向上传播
3. 直到某个 try/catch 捕获
4. 或者到最顶层

**这叫：**

> 控制流向上传播

产生错误的代码：不再决定处理方式。

## 讲义 Placement 例子（Page 65）

正确写法：

```java
Placement(String descr) {  
    if (error) {  
        throw new IllegalArgumentException("Invalid placement");  
    }  
}
```

这里没有：

- 打印
- exit
- 重试

只是 " 报告问题 "。

## 错误处理在更高层（Page 66）

```java
Placement playTurn() {  
    while (p == null) {  
        try {  
            p = readPlacement(input);  
        }  
        catch(IllegalArgumentException iae) {  
            output.println(iae.getMessage());  
            output.println("Please try again");  
        }  
    }  
}
```

现在：

- 构造函数负责检测错误
- Controller 决定如何处理

——这是解耦。

## 异常传播机制的关键理解

假设调用栈：

```
main  
  ↓  
playTurn  
  ↓  
readPlacement  
  ↓  
new Placement
```

Placement 抛出异常：

- readPlacement 没 catch → 继续往上
- playTurn catch → 处理
- main 不知道

这叫：

> 自动传播

## 在 Java 中测试异常：`assertThrows`（Page 67–75）

`assertThrows` 是 Java 单元测试（尤其是 **JUnit 5**）中非常重要的一个工具方法。

它的核心作用是：

> **断言某段代码会抛出指定类型的异常。**

### assertThrows 是做什么的？

在单元测试中，我们通常测试两种情况：

1. 正常情况（valid case）
2. 错误情况（error case）

例如：

```java
new Placement("A1H");   // 合法  
new Placement("A0");    // 非法，应抛异常
```

对于错误情况，我们希望测试：

> 构造函数是否真的抛出了 IllegalArgumentException？

这时就用：

```java
assertThrows(IllegalArgumentException.class,  
             () -> new Placement("A0"));
```

###基本语法

JUnit 5 的定义是：

```java
<T extends Throwable>  
T assertThrows(Class<T> expectedType,  
               Executable executable)
```

两个参数：

|参数|作用|
|---|---|
|expectedType|期望的异常类型|
|executable|可能抛异常的代码|

#### 示例

```java
@Test  
void test_invalidPlacement() {  
    assertThrows(IllegalArgumentException.class,  
                 () -> new Placement("A0"));  
}
```

测试通过的条件是：

- 代码抛出了 IllegalArgumentException
- 或其子类

### 为什么要用 lambda？

讲义特别强调这一点（Page 71–72）。很多人会写错：

```java
Placement p = new Placement("A0");   // ❌ 异常在这里就抛出  
assertThrows(IllegalArgumentException.class, p);
```

问题：异常在 assertThrows 调用之前已经发生。assertThrows 根本没机会捕获。

#### 正确方式

必须把可能抛异常的代码：

> 包裹在一个 " 延迟执行 " 的函数里。

这就是 lambda 的作用：

```java
() -> new Placement("A0")
```

这段代码不会立即执行。它只是传给 assertThrows。assertThrows 内部才调用它。

### 内部原理（讲义 Page 74）

讲义给出了核心逻辑：

```
try {  
    executable.execute();  
    fail("Did not throw exception");  
}  
catch (Throwable t) {  
    check type;  
}
```

解释一下：

1. 执行 lambda
2. 如果没抛异常 → 测试失败
3. 如果抛异常 → 检查类型
4. 如果类型匹配 → 测试通过

### `assertThrows` 返回值

很多人不知道：

> assertThrows 会返回捕获到的异常对象。

例如：

```java
IllegalArgumentException ex =  
    assertThrows(IllegalArgumentException.class,  
                 () -> new Placement("A0"));  
  
assertEquals("Invalid placement", ex.getMessage());
```

你可以测试：

- 异常类型
- 异常 message

### 常见错误

#### 错误 1：提前执行

```java
new Placement("A0");  
assertThrows(…);
```

#### 错误 2：捕获了异常

```java
try {  
    new Placement("A0");  
}  
catch(…) {  
}
```

assertThrows 永远不会看到异常。

#### 错误 3：抛错类型不对

```java
assertThrows(RuntimeException.class, …);
```

如果实际抛的是 NullPointerException，测试仍然通过（因为是子类）。这有时不是你想要的。

## 总结

### 为什么异常设计是解耦？

现在我们总结一下。

**551 方式：**

```
Placement → exit
```

错误产生 = 错误处理——强耦合 ❌

**651 方式：**

```
Placement → throw  
Controller → catch  
main → maybe exit  
Test → assertThrows
```

错误产生 ≠ 错误处理——低耦合 ✔

### 什么时候可以 exit？

讲义 Page 81 说：

> Main? Ok.
> Depths of library? Likely a bad idea.

换句话说：**越底层代码，越不能 exit。**

### 异常解耦的核心思想

异常是：

> 一种控制流机制
> 让错误处理向上移动

也可以说：

> 把 " 处理决策权 " 从底层抽离出来。

### 和前面 MVC 的联系

异常解耦本质是：

> Model 不负责 UI
> 也不负责流程控制

**Model 只负责：**

- 维护不变量
- 抛出错误

**Controller 负责：**

- 决定是否重试
- 决定是否退出

# 数据差异 vs 行为差异 (Page 83–90)

这是面向对象设计中一个非常高级的思想。问自己：

> 不同对象是 " 数据不同 "？
> 还是 " 行为不同 "？

Battleship 例子：Submarine / Destroyer / Carrier

只差：

- 长度
- 字母
- 名字

👉 只是数据不同

👉 可以用一个 RectangleShip 类

后来不规则船只要把它们表示为：

```
Set<Coordinate>
```

👉 差异仍然是 " 数据 "

目标：

> 尽量把差异转化为数据
> 而不是写新的类和行为
