# 1 软件工程的目标与现实约束（Goals & Trade-offs）

## 1.1 用户 / 客户想要什么样的软件？

讲义一开始并没有直接给答案，而是通过 **Think–Pair–Share** 让大家先思考这个问题（第 3 页）：

> **"What characteristics of software would customers desire?"**

随后，讲义给出了一个**非常典型、也非常现实的列表**（第 4–5 页）

![[Pasted image 20260111112855.png|500]]

### 1.1.1 软件的常见目标（Software Goals）

讲义列出的目标包括：

1. **Correct（正确性）**
	- 软件是否 " 做了它应该做的事 "
	- 没有 bug 👉 *这是底线（must-have）*
2. **Robust（健壮性）**
	- 安全（secure）
	- 可靠（dependable）
	- 对异常情况有弹性（resilient）
3. **Usable（可用性）**
	- 好的 UI
	- 兼容性
	- 用户体验
4. **Many features（功能多）**
	- 能做 " 很多有用的事 "
5. **Good performance（性能）**
	- 快
	- 省电
	- 高效利用资源
6. **Scalable（可扩展性）**
	- 用户多了还能撑得住
	- 数据大了还能跑
7. **Maintainable / Extensible（可维护 / 可扩展）**
	- 将来能加功能
	- 能修 bug 而不 " 牵一发动全身 "
8. **Quick delivery（快速交付）**
	- 尽快上线
	- 抢市场窗口
9. **Low cost（低成本）**
	- 少花钱
	- 少人力
	- 少时间

> ⚠️ 讲义特别强调：
> **有些目标是 " 必须的 "（如 Correct），
> 有些是 " 模糊且可权衡的 "。**

## 1.2 这些目标能否同时满足？

==**答案：不能。**==

讲义在第 4 页就明确点出：

> **"Of these, which work well together, which are in opposition?"**
> ——有些目标彼此支持，有些目标天然冲突 

### 1.2.1 举几个典型 " 互相冲突 " 的例子

|目标 A|目标 B|冲突原因|
|---|---|---|
|功能多|快速交付|功能越多，开发时间越长|
|高性能|可维护性|高度优化代码往往难读难改|
|低成本|高质量|少人、少时间 → 更容易出 bug|
|快速交付|健壮 / 安全|来不及设计、测试|

👉 **软件工程的核心不是 " 全部最大化 "，而是 " 有意识地取舍 "。**

## 1.3 典型三角：低成本 / 快速交付 / 大规模功能

讲义在第 6–7 页给出了一个非常经典的模型（你可以把它当成 " 软件工程版不可能三角 "）

![[Pasted image 20260111113052.png|400]]

- 这三个目标 **两两之间都存在冲突**
- **你不能同时拥有三个**
- 软件工程的作用是：

> **" 支持更多其中的两个，但永远不可能全部 "**

换句话说：

|你想要|你必须牺牲|
|---|---|
|快 + 大|钱|
|快 + 便宜|功能|
|大 + 便宜|时间|

## 1.4 为什么现实中不能 " 三者兼得 "？

讲义没有停留在 " 口号 "，而是用一个**非常工程化的反例**来解释（第 8–10 页）

### 1.4.1 天真的想法：人多就能快

讲义假设：

- 10 个开发者
- 1 年完成
- 每人年薪 $125,000
- 总成本 $1,250,000

然后提出一个**看似合理的问题**：

> 那我雇 20 个人，6 个月做完，不就又快又不贵吗？

甚至继续极端化：

- 40 人 → 3 个月？
- 3,650 人 → 1 天？
- 5,256,000 人 → 1 分钟？

### 1.4.2 关键转折：软件不是 " 可尴尬并行 " 的问题

讲义在第 9–10 页给出了一个**非常重要的概念**：

> **Software development is NOT embarrassingly parallel** 

#### 1.4.2.1 什么是 embarrassingly parallel？

- 任务可以几乎 **完全独立**
- 几乎不需要沟通
- 比如：渲染不同图片、计算不同数据块

#### 1.4.2.2 软件开发的问题是

- 模块之间高度关联
- 接口必须对齐
- 需要频繁沟通、同步、协调

### 1.4.3 Brooks 定律（人月神话）

讲义给出了 Frederick P. Brooks 的经典结论（第 10–11 页）

> **Adding manpower to a late software project makes it later**

原因包括：

- 新人需要培训
- 沟通成本呈 **O(n²)** 增长
- 决策变慢
- 合并冲突、设计分歧变多

所以现实情况往往是：

|开发人数|完成时间|
|---|---|
|10 人|12 个月|
|20 人|9 个月|
|40 人|8 个月|

👉 **远远不是线性加速**

> **软件工程不是追求 " 完美的软件 "，
> 而是在不可避免的约束下，
> 用好的设计，把代价降到最低。**

- 引出一个关键结论：

> **要想并行开发，必须把系统拆成真正 " 独立 " 的模块**

👉 这直接引出后面**设计原则的技术动机**：**好设计 = 降低人和人之间的耦合**

# 2 面向对象设计的四大基石

**OO 四大基石本身并不等于好设计，好设计是 " 用对了 " 它们。**

## 2.1 Inheritance（继承）——代码复用 ≠ 设计复用

**表面理解（新手常见）**

- 子类自动获得父类的字段和方法
- 少写代码

**设计层面的真正作用**

- **建立 "is-a" 语义关系**
- 为 **多态（Polymorphism）** 提供基础
- 让系统可以 **在不改旧代码的情况下扩展**

**在设计原则中的角色**

- **Open/Closed Principle（OCP）** 的基础
- **Liskov Substitution Principle（LSP）** 的约束对象

👉 讲义后面用 Penguin / Bird 反复强调：**继承本身很容易用错，是 " 最危险的 OO 工具 "。**

## 2.2 Polymorphism（多态）——消灭 if/else 的核心武器

**表面理解**

- 父类指针 / 引用指向子类对象
- 调用虚函数时动态分派

**设计层面的真正作用**

- **把 " 变化 " 推迟到运行时**
- 用 " 对象行为差异 " 代替：

```cpp
if (type == A) { … }
else if (type == B) { … }
```

**在设计原则中的角色**

- **Open/Closed Principle（OCP）** 的直接实现手段
- 降低修改成本
- 提高可扩展性

👉 讲义非常直白地说：

> **if / else / switch 是 Open/Closed 的敌人** 

## 2.3 Abstraction（抽象）——决定 " 依赖什么，而不依赖什么 "

 **表面理解**

- 抽象类
- 接口（Java interface / C++ abstract class）

**设计层面的真正作用**

- **隐藏不重要的细节**
- **稳定依赖方向**
- 让高层模块不依赖具体实现

**在设计原则中的角色**

- **Dependency Inversion Principle（DIP）**
- **Interface Segregation Principle（ISP）**
- 支撑模块化、可测试性

👉 抽象的核心不是 " 更高级 "，而是：

> **" 把可能变化的部分隔离出去 "**

## 2.4 Encapsulation（封装）——控制变化的 " 传播半径 "

**表面理解**

- private / protected / public
- getter / setter

**设计层面的真正作用**

- **限制谁能看到什么**
- 防止 " 无意的依赖 "
- 降低修改引发连锁反应的概率

**在设计原则中的角色**

- **Least Surprise**
- **Low Coupling**
- **Design for Testability**

👉 封装的目的不是 " 藏起来 "，而是：

> **让错误更难发生，让修改影响更小**

# 3 设计原则

## 3.1 设计原则清单

- Least Surprise / Astonishment（最小惊讶原则）
- DRY（Don't Repeat Yourself）
- Low Coupling / High Cohesion（低耦合 / 高内聚）
- **SOLID 原则**
	- S：Single Responsibility
	- O：Open / Closed
	- L：Liskov Substitution
	- I：Interface Segregation
	- D：Dependency Inversion
- Design for Testability（为测试而设计）

## 3.2 三个 " 基础但极其重要 " 的设计原则

### 3.2.1 Least Surprise / Astonishment（最小惊讶原则）

**定义（讲义原意）**

> **减少其他开发者在使用你的代码时感到 " 惊讶 " 的程度**

**具体体现**

- 方法名符合直觉
- 函数不产生隐藏副作用
- 不依赖 " 你必须知道的隐含顺序 "

**讲义中的典型反例**

- `getSomething()` 改变了对象状态
- " 你必须先调用 A，再调用 B，否则程序崩溃 "
- " 传负数会死循环，但文档没写 "

**设计层面的意义**

- 降低认知负担
- 减少误用
- 降低维护成本

👉 如果一个功能 " 必须令人惊讶 "，那说明：

> **设计可能本身有问题**

### 3.2.2 DRY（Don't Repeat Yourself）

**定义**

> **系统中每一条知识都应该有唯一、权威的表达**

**常见表现**

- 重复的逻辑
- 重复的常量
- 复制粘贴代码

**讲义给出的 " 自检标准 "**

> 如果你改了 A 处代码，就必须在 B 处做同样的改动
> **那你已经违反 DRY 了** 

**==⚠️ 讲义的重要提醒==**

- **不要过早抽象（DAMP）**
- 有些重复是可以接受的：
	- 提高可读性
	- 保持简单
- 当 " 变化需求出现 " 时再重构

👉 DRY ≠ " 一切都抽象成函数 "

### 3.2.3 Low Coupling / High Cohesion（低耦合 / 高内聚）

这是**所有后续设计原则的 " 母原则 "**。

#### 3.2.3.1 Coupling（耦合）

> 模块之间的相互依赖程度

- 高耦合：改一个类，很多类要跟着改
- 低耦合：模块之间通过清晰接口通信

#### 3.2.3.2 Cohesion（内聚）

> 一个模块内部的功能相关程度

- 高内聚：一个类 / 模块只做 " 相关的事情 "
- 低内聚：什么都往一个类里塞

## 3.3 SOLID 原则

**SOLID 不是 5 条 " 规则 "，而是一套 " 设计约束体系 "**：

| 字母  | 原则                    | 解决的核心问题     |
| --- | --------------------- | ----------- |
| S   | Single Responsibility | 一个模块为什么会变化  |
| O   | Open / Closed         | 如何在不改旧代码下扩展 |
| L   | Liskov Substitution   | 继承是否 " 真的合理 "  |
| I   | Interface Segregation | 接口是否在强迫依赖   |
| D   | Dependency Inversion  | 依赖方向是否正确    |

### 3.3.1 S—Single Responsibility Principle（单一职责原则, SRP）

#### 3.3.1.1 原则定义（讲义版）

> **A class/module should have only one reason to change**

也就是：

> **一个模块只应该因为 " 一种原因 " 而变化**

⚠️ 注意：

- **不是**" 只能有一个方法 "
- **不是**" 只能干一件小事 "
- 而是：**只对一种变化负责**

#### 3.3.1.2 讲义的核心判断标准：变化原因（Reason to Change）

讲义反复强调一个判断方法（飞机、DataParser 例子）：

> **如果不同需求变化会导致修改同一个类 → 违反 SRP**

#### 3.3.1.3 典型反例（讲义中的飞机类）

一个 `Airplane` 类里同时包含：

- 路线规划
- 发动机控制
- 通信（ATC）

表面上看：

> " 这都是飞机做的事 "

但讲义问了一个**工程化问题**：

- 如果发动机型号变了 → 哪些代码该改？
- 如果通信协议变了 → 哪些代码该改？
- 如果路线算法升级 → 哪些代码该改？

👉 **答案指向不同变化源 → 职责不同**

#### 3.3.1.4 为什么 SRP 如此重要？

1. 降低修改风险: 改 A 功能，不会破坏 B 功能
2. 提高可维护性: " 改哪里 " 一眼就知道
3. 提升可测试性（后面会反复出现）: 一个职责 → 一类测试

#### 3.3.1.5 SRP 与 OO 基石的关系

|OO 概念|在 SRP 中的作用|
|---|---|
|Encapsulation|把变化限制在最小范围|
|Abstraction|把不同职责隔离|
|Polymorphism|后续用于解耦不同变化|

### 3.3.2 O—Open / Closed Principle（开闭原则，OCP）

#### 3.3.2.1 原则定义（讲义原话）

> **Software entities should be open for extension, but closed for modification**

翻译成人话：

> **添加新功能时，尽量不改已有代码**

#### 3.3.2.2 为什么 " 修改旧代码 " 是危险的？

讲义给了非常现实的原因：

- 修改代码可能引入新 bug
- 你可能：
	- 没有全部上下文
	- 甚至没有源码（第三方库）

👉 **工程实践中，" 不动旧代码 " 本身就是价值**

#### 3.3.2.3 讲义中的经典反例：if / else / switch

讲义明确说：

> **If/else or switch are the enemies of Open/Closed**

原因很简单：

```cpp
if (type == A) { … }
else if (type == B) { … }
else if (type == C) { … }
```

- 每加一个新类型
- 就必须修改这段代码
- → **违反 OCP**

#### 3.3.2.4 讲义给出的 " 正确解法 "：多态

```cpp
class Bird {
  virtual void fly() = 0;
};

class Pigeon : public Bird { … };
class Parrot : public Bird { … };
class Hawk   : public Bird { … };
```

- 新增鸟类 → 新增类
- 旧代码完全不动

👉 **OCP = Inheritance + Polymorphism 的正确使用**

### 3.3.3 L—Liskov Substitution Principle（里氏替换原则，LSP）

这是 **SOLID 中最容易被 " 写代码的人忽视 "，但最重要的原则之一**。

#### 3.3.3.1 原则定义（讲义版）

> **Objects of a superclass should be replaceable with objects of a subclass without breaking correctness**

也就是说：

> **只要代码在用父类，它就必须能安全地使用任何子类**

#### 3.3.3.2 讲义的核心反例：企鹅（Penguin）

- `Bird` 有 `fly()`
- `Penguin` 继承 `Bird`
- 但企鹅不会飞

程序运行：

- 用户点击 " 飞 "
- 企鹅忽略 / 乱动

👉 **虽然代码能编译，但行为违背了 "Bird 的契约 "**

#### 3.3.3.3 讲义强调：LSP ≠ " 能继承就行 "

LSP 引入了更严格的**行为约束**，子类必须：

- **不加强前置条件**
- **不削弱后置条件**
- **保持父类不变量**
- **保持历史一致性**

讲义用了：

- `LinkedList` / `WeirdList`
- 前置条件、后置条件、不变量来说明这些概念

#### 3.3.3.4 LSP 的工程意义

> **如果你的继承层次违反 LSP，
> 那你的多态就是 " 伪多态 "。**

后果包括：

- 隐蔽 bug
- 不可预测行为
- 单元测试难写

### 3.3.4 I—Interface Segregation Principle（接口隔离原则，ISP）

#### 3.3.4.1 原则定义（讲义版）

> **A client should not be forced to depend on methods it does not use**

#### 3.3.4.2 讲义的核心问题

当一个接口包含：

- 太多方法
- 不同客户端只用其中一部分

就会出现：

- " 被迫实现不需要的方法 "
- 抛异常 / 空实现

👉 这是 **ISP 的典型违反信号**

#### 3.3.4.3 讲义的 DataSource 反例

`SyntheticDataSource`：

- 只想 `parseData`
- 却被迫实现：
	- `getSourceAddress`
	- `getInputStream`

最终结果：

```cpp
throw some_exception("not appropriate");
```

👉 **接口设计失败**

#### 3.3.4.4 ISP 的设计目标

- 多个小接口
- 而不是一个 " 万能接口 "

### 3.3.5 D—Dependency Inversion Principle（依赖反转原则，DIP）

这是 **SOLID 中最 " 架构级 " 的原则**。

#### 3.3.5.1 原则定义（讲义版）

> **High-level modules should not depend on low-level modules.
> Both should depend on abstractions.**

#### 3.3.5.2 讲义的 Manager / Employee 例子

**错误设计**

- `Manager` 直接创建 `Developer`, `Tester`
- 强耦合
- 新员工类型 → 改 Manager

**正确设计**

- `Manager` 只依赖 `Employee` 抽象
- 员工通过 **Dependency Injection** 传入

#### 3.3.5.3 DIP 与 [[Dependency Injection (DI)]] 的关系

讲义明确区分：

- **DIP**：设计原则（目标）
- **DI**：实现方式（手段）

> **DI 是实现 DIP 的常见方式**

## 3.4 Design for Testability（为测试而设计，DFT）

### 3.4.1 什么是 Design for Testability（为测试而设计）？

讲义在最后并没有说 " 测试很重要 "，而是说：

> **测试性不是测试阶段才考虑的事，
> 而是设计阶段就决定的属性。**

也就是说：

- ❌ 不是：" 代码写完了，想办法测 "
- ✅ 而是：" 代码这样设计，是不是**本来就好测**？"

### 3.4.2 什么叫 " 好测试的代码 "？

讲义给了一个非常工程化的判断方式（第 83 页）：

> **测试代码是否能：**
>
> - 控制被测模块的状态
> - 观察被测模块的结果
> - 把它从系统中隔离出来

这三点就是下面的三个指标。

### 3.4.3 测试性的三大指标（讲义原文拆解）

#### 3.4.3.1 Controllability（可控性）

> **测试代码能否方便地控制被测模块的输入和状态？**

##### 3.4.3.1.1 好的可控性意味着

- 可以注入：
	- 假数据
	- 假依赖（Fake / Mock）
- 可以构造：
	- 边界条件
	- 异常情况

##### 3.4.3.1.2 可控性差的典型例子

```cpp
class Service {
  Database db; // 写死
};
```

- 测试者无法：
	- 换数据库
	- 注入错误
	- 模拟超时

👉 **没有 Dependency Injection = 没有可控性**

#### 3.4.3.2 Observability（可观测性）

> **测试代码能否 " 看见 " 模块的行为是否正确？**

##### 3.4.3.2.1 好的可观测性意味着

- 有明确的返回值
- 有可检查的状态
- 行为可通过接口观察

##### 3.4.3.2.2 可观测性差的例子

- 只打印日志，不返回结果
- 通过全局变量 " 偷偷 " 修改状态
- 隐藏副作用

👉 这也是 **Least Surprise 原则**的延伸。

#### 3.4.3.3 Isolatability（可隔离性）

> **能否把被测模块 " 单独拿出来测试 "？**

这是 **单元测试的前提条件**。

##### 3.4.3.3.1 不可隔离的典型特征

- 一个类：
	- 读文件
	- 连网络
	- 解析数据
	- 计算结果
- 测它 = 启动整个系统

👉 这正是 **违反 SRP 的直接后果**。

## 3.5 Design for Testability 与 SRP 的关系（讲义重点）

讲义在最后一页直接抛出问题：

> **HOW DOES FOLLOWING THE SINGLE RESPONSIBILITY PRINCIPLE HELP WITH TESTING?**

我们来把这个问题彻底讲清楚。

### 3.5.1 SRP → Isolatability（最直接的关系）

#### 3.5.1.1 SRP 的核心

> **一个模块只有一个变化原因**

这意味着：

- 功能单一
- 依赖更少
- 职责边界清晰

👉 结果是：

> **这个模块可以被单独拿出来测**

### 3.5.2 对比示例

#### 3.5.2.1 违反 SRP 的类

```cpp
class DataProcessor {
  void readFile();
  void parseData();
  void compute();
  void writeOutput();
};
```

- 测 `compute()`：
	- 必须准备文件
	- 必须跑解析
	- 必须处理输出

👉 **测试成本爆炸**

#### 3.5.2.2 遵守 SRP 的设计

```cpp
class DataReader { … };
class DataParser { … };
class Calculator { … };
class OutputWriter { … };
```

- 测 `Calculator`：
	- 输入一个数据结构
	- 直接验证输出

👉 **天然单元测试友好**

### 3.5.3 SRP → Controllability

SRP 往往意味着：

- 依赖更少
- 更容易注入假对象

例如：

```cpp
class Calculator {
public:
  int compute(const Data& d);
};
```

而不是：

```cpp
class Calculator {
  Database db;
};
```

👉 **控制输入 = SRP 的自然结果**

### 3.5.4 SRP → Observability

SRP 的类：

- 方法语义清晰
- 返回值明确
- 副作用少

测试代码更容易：

- 判断 " 对 / 错 "
- 精确定位 bug

## 3.6 从设计原则到测试技术的 " 因果链 "

这是 ECE651 非常希望你建立的整体认知：

```
SRP
 ↓
低耦合 / 高内聚
 ↓
Dependency Injection
 ↓
高可控性 / 可隔离性
 ↓
可单元测试
 ↓
高质量软件
```

👉 **测试不是 " 补救措施 "，而是 " 设计质量的副产品 "。**

> **如果一段代码很难测试，
> 那它很可能已经违反了某条设计原则。**

不是因为你 " 不会写测试 "，而是设计已经出问题了。
