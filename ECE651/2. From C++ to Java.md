# 1 Java 与 C++ 的差异和新特性

> Java 在 " 表面语法 " 上极力降低 C++ 程序员的学习成本，但在 " 语义层面 " 做了大量强制性的设计取舍。

## 1.1 基本类结构（Page 3–6）

### 1.1.1 类、字段、构造函数、方法：为什么你会觉得 " 很熟悉 "

看 Page 3 的 `Point` 示例，几乎每一行都能在 C++ 里找到影子：

```java
public class Point {
  private final int x;
  private final int y;

  public Point() {
    this(0, 0);
  }

  public Point(int x, int y) {
    this.x = x;
    this.y = y;
  }

  public int getX() { return x; }
  public int getY() { return y; }
}
```

**对应的 C++ 心智模型：**

|Java|C++|
|---|---|
|class|class|
|private / public|private / public|
|构造函数同名|构造函数同名|
|方法写在类里|方法写在类里|

👉 **这不是巧合**：Java 的设计目标之一就是——

> *" 让 C++ 程序员第一天就能写出可工作的代码 "*

### 1.1.2 关键差异一：**所有对象都是 " 引用语义 "**

这是 **Java 和 C++ 最大的根本差异之一**。

#### 1.1.2.1 在 C++ 中

```cpp
Point p(3,4);      // 栈上对象
Point* q = &p;    // 指针
```

- `.` vs `->`：**语法上区分**
- 你能一眼看出：
	- 是对象
	- 还是指针
- 也因此会出现：
	- 悬空指针
	- double free
	- use-after-free

---

#### 1.1.2.2 在 Java 中

```java
Point p = new Point(3,4);
```

**关键事实：**

- `Point` 是对象类型
- `p` **永远是一个引用**
- 没有 " 值语义的对象 "

所以：

```java
p.getX();   // 一定是“通过引用”
```

👉 **设计含义（非常重要）**：

- Java 牺牲了一部分性能与控制力
- 换来了：
	- 统一的语义
	- 极大减少的内存错误
	- 更强的抽象

这是典型的 **Least Surprise + Safety over Power**。

---

### 1.1.3 没有 `* / ->`，只有 `.`

这不是 " 语法糖 "，而是**强制统一模型**：

- Java 不让你区分：
	- " 我是在用对象 "
	- " 我是在用指向对象的指针 "
- 因为在 Java 看来：

	> **" 对象 ≈ 引用 " 是唯一模型**

👉 这一步直接**消灭了一整类错误**，但也**消灭了一整类表达能力**。

### 1.1.4 没有 initializer list

#### 1.1.4.1 C++

```cpp
Point(int x, int y) : x(x), y(y) {}
```

#### 1.1.4.2 Java

```java
public Point(int x, int y) {
  this.x = x;
  this.y = y;
}
```

Java 的选择：

- 所有初始化都是**赋值**
- 没有 " 先初始化、后进入构造体 " 的机制

👉 代价：

- 无法用 initializer list 表达复杂不变式

👉 好处：

- 构造过程更 " 线性、直观 "
- 减少微妙的初始化顺序 bug

### 1.1.5 `final ≈ const`，但**语义不同（先埋个雷）**

```java
private final int x;
```

你现在可以先记住一句话（后面会展开）：

> **Java 的 `final` 更接近 " 不可重新赋值 "，而不是 C++ 的 " 不可修改对象 "。**

这是后面：

- 不可变对象
- 并发
- 设计契约

的关键基础。

## 1.2 `toString()` / `equals()` / `hashCode()`（Page 6–25）

这是整讲的**灵魂部分**，也是你以后写 Java 最容易踩雷的地方。

### 1.2.1 `toString()`：为什么打印出来是 `Point@4b1210ee`

#### 1.2.1.1 发生了什么？

```java
System.out.println(p);
```

Java 实际做的是：

```java
System.out.println(p.toString());
```

而如果你没 override：

```java
Object.toString(): //类名 + @ + 十六进制 hash
```

👉 对调试几乎**毫无价值**

#### 1.2.1.2 正确做法

```java
@Override
public String toString() {
  return "(" + x + ", " + y + ")";
}
```

**这里已经出现两个设计思想：**

1. `@Override`：编译期帮你防错
2. Java 不允许你乱重载操作符，但**给你 string 拼接**

### 1.2.2 `==` vs `equals()`：两种 " 相等 "

这是**C++ 程序员最容易犯错的地方**。

```java
Point p1 = new Point(3,4);
Point p2 = new Point(3,4);

p1 == p2        // false
p1.equals(p2)  // true（如果你写对了）
```

#### 1.2.2.1 含义区分（务必牢记）

|表达式|含义|
|---|---|
|`==`|是否同一个对象（同一引用）|
|`equals()`|是否语义相等|

👉 `==` 永远不应该被重载

👉 `equals()` 是你**对 " 相等 " 的定义**

### 1.2.3 `equals()` 的契约（Contract）

Java 明确规定：你写的 `equals()` **必须满足**：

1. **Reflexive** - `x.equals(x)` 必须是 true
2. **Symmetric** - `x.equals(y)` ⇔ `y.equals(x)`
3. **Transitive** - `x == y && y == z ⇒ x == z`
4. **Consistent** - 对象状态不变，结果不能变
5. **Non-null** - `x.equals(null)` 必须是 false

👉 这是**数学意义上的等价关系**。

### 1.2.4 继承 + equals 的陷阱（最容易出大事）

#### 1.2.4.1 天真写法（很多 IDE 会给你生成）

```java
if (other instanceof Point) {
  Point p = (Point) other;
  return x == p.x && y == p.y;
}
```

**问题在哪里？**

引入子类：

```java
class Point3D extends Point {
  private int z;
}
```

```java
Point p  = new Point(3,4);
Point3D q = new Point3D(3,4,7);

p.equals(q)  // true
q.equals(p)  // false
```

🚨 **直接违反 Symmetry**

#### 1.2.4.2 正确（但更严格）的做法

```java
if (other != null && other.getClass().equals(getClass())) {
  …
}
```

👉 这会让：

- `Point` 只等于 `Point`
- `Point3D` 只等于 `Point3D`

**设计取舍**：

- ✔ 保证数学正确性
- ❌ 放弃 " 跨类型相等 "

### 1.2.5 `hashCode()`：为什么 equals 和 hashCode 必须一起写

Java 容器（`HashMap`, `HashSet`）的前提：

> **如果 `a.equals(b)`，那么 `a.hashCode() == b.hashCode()`**

否则：

- 查不到元素
- 出现幽灵 bug

👉 **这是工程灾难级别的错误**

## 1.3 `halfEquals`：当语言不够，用设计模式补

现在问题来了：

> " 如果我真的想让 `Point(3,4)` 等于 `Point3D(3,4,0)` 呢？"

Java 没有给你现成机制。

### 1.3.1 解决方案：**双向半相等（halfEquals）**

核心思想：

- 把 " 是否相等 " 的逻辑拆成两个方向
- 双方都同意，才算相等

```java
protected boolean halfEquals(Point p) {
  return x == p.x && y == p.y;
}

@Override
public boolean equals(Object other) {
  if (!(other instanceof Point)) return false;
  Point p = (Point) other;
  return this.halfEquals(p) && p.halfEquals(this);
}
```

子类只 override `halfEquals`。

### 1.3.2 这段设计的**软件工程价值**

这正是 ECE 651 想教你的：

> **当语言机制无法同时满足所有设计目标时：**
>
> - 不要 "hack"
> - 不要破坏契约
> - 用清晰的设计模式补上

# 2 对象与基本类型

## 2.1 Java 的两类 " 值 "：Primitive vs Object（包装类）

### 2.1.1 Primitive（8 种基本类型）

讲义列了 8 个 primitive：`boolean, byte, char, double, int, float, long, short`（还有 `void` 表示 " 没有值 "） 

**特征：**

- 变量里放的是**值本身**（bit pattern）
- 不为 null
- 不参与继承体系（不是 Object 的子类）
- 运算通常是 CPU 直接算（更接近 C++ 的内建类型）

### 2.1.2 Object（引用类型）与 Wrapper（包装类）

每个 primitive 都有对应的 " 包装类 "：

- `int` ↔ `Integer`
- `double` ↔ `Double`
- `boolean` ↔ `Boolean`
- ……

**特征：**

- 变量里放的是**引用（reference）**，指向堆上的对象
- 可以为 null（危险点：`NullPointerException`）
- 有方法/接口/泛型能力（能放进 `ArrayList<Integer>`、`HashMap<Integer, …>`）

> 关键：Java 的集合/泛型只能放 Object（确切说是引用类型），放不了 primitive。
> 所以你在集合里看到的 `Integer`，不是 `int`。

## 2.2 内存差异：为什么 primitive 更省

讲义在 Page 24 明确问："Which one is more memory efficient?" 并对比 `int a = 42;` 和 `Integer b = 42;`

### 2.2.1 直觉结论

- `int`：只存 4 字节（32-bit）值（外加栈帧/对齐等，但概念上很小）
- `Integer`：至少包含
	
	1. 一个引用（在 64-bit JVM 上通常 4 或 8 字节，取决于压缩指针）
	2. 堆上的 `Integer` 对象本体：对象头 + 一个 int 字段 + 对齐填充

所以 `Integer` 的 " 总成本 " 往往是 `int` 的好几倍，甚至十几倍。

### 2.2.2 工程含义（非常实用）

- 大量数值数据（比如百万级）：`int[]` 通常比 `ArrayList<Integer>` **省内存、快很多**
- 这会直接影响：
	- GC 压力（对象多 → GC 更频繁）
	- 缓存友好性（连续数组 vs 零散对象）

## 2.3 性能差异：为什么 primitive 更快

讲义同一页也问："Which one is faster?" 并对比 `int a = 1 + 2;` 与 `Integer b = 1 + 2;` 

### 2.3.1 原因拆解

`int` 的 `1+2`：

- 直接 CPU 加法指令
- 无分配、无间接访问

`Integer b = 1 + 2` 表面很像，但背后通常涉及：

- **装箱（boxing）**：把 primitive 变成对象（可能分配对象，或使用缓存）
- **拆箱（unboxing）**：把对象里的 int 值取出来再运算
- 可能触发 null 检查（拆箱时若为 null 会 NPE）

> [!NOTE] 一句话记忆
> - primitive：算值
> - wrapper：可能 " 拆箱→算→装箱 "（还可能分配对象）

## 2.4 参数传递语义：Java 只有一种——" 按值传递 "（但值可能是引用）

这是理解经典陷阱的核心：

> Java **永远是 pass-by-value（按值传递）**。
> 只是这个 " 值 " 如果是对象变量，那它的值是 " 引用 "。

## 2.5 经典陷阱示例：`int` vs `Integer`（Page 24 的代码）

讲义给了两个函数（一个改 int，一个改 Integer），然后问 main 里打印什么。 

我们用 " 语义 " 把它讲透（不需要记答案，记机制）。

### 2.5.1 `int` 版本：改不到外面

```java
public static void func_int(int a) {
  a = 100;
}
```

调用时：

```java
int arg1 = 10;
func_int(arg1);
```

- 传进去的是 `arg1` 的值 `10`
- 形参 `a` 是一个新的局部变量
- `a=100` 只改了局部副本
- `arg1` 仍然是 10

✅ 这和 C/C++ 的 " 传值 " 完全一致。

### 2.5.2 `Integer` 版本：看起来像 " 引用 "，但还是改不到外面

```java
public static void func_Integer(Integer a) {
  a = 100;
}
```

调用：

```java
Integer arg2 = 10;
func_Integer(arg2);
```

这里最容易误会：很多人以为 " 对象是引用语义，所以能改到外面 "。

但注意：**传进去的是引用的副本**。

- `arg2` 保存的是一个引用（指向某个 `Integer(10)` 对象）
- 调用时，把这个引用值复制一份给形参 `a`
- `a = 100` 的含义是：让形参 `a` 指向另一个 `Integer(100)` 对象（或缓存对象）
- 这并不会改变外面的 `arg2` 仍指向原来的 `Integer(10)`

所以外面还是 10。

> 这就是你提到的那句话：
> **`Integer` 传的是 " 引用的拷贝 "，不是 " 引用本身 "。**

## 2.6 " 那怎样才能改到外面？"——两种情况要区分

### 2.6.1 情况 A：你 " 重新绑定引用 " → 改不到外面

```java
void f(Point p) { p = new Point(…); } // 外面不变
void g(Integer a){ a = 100; }           // 外面不变
```

### 2.6.2 情况 B：你 " 通过引用修改对象内部状态 " → 能影响外面（如果对象可变）

```java
void h(List<Integer> xs) { xs.add(1); } // 外面看得到
```

关键条件是：对象本身要**可变（mutable）**，且你在函数里做的是 " 改对象内部 "，不是 " 换指向 "。

## 2.7 为什么这为 " 不变对象 / 副作用 / 可测试性 " 铺垫？

### 2.7.1 不变对象（Immutable）让推理更简单

像 `String`、`Integer` 这类通常是不可变的（你 " 改它 " 其实是创建新对象）。

结果：

- 函数里即便拿到引用，也没法改内部状态
- 更少隐式副作用
- 更容易写出可预测的代码

这对测试特别友好：

**输入是什么 → 输出就稳定是什么**（更接近纯函数）。

### 2.7.2 副作用（Side effect）是测试的敌人

如果你传进去的是一个可变对象，比如 `ArrayList`，函数里 `add/remove` 就会改变外部世界。

- 好处：方便
- 坏处：测试时要额外验证 " 对象是否被改了 "，还要准备/清理状态

所以软件工程里经常会强调：

- 尽量用不可变对象
- 或明确标注/限制可变对象的修改范围（封装、接口隔离）

### 2.7.3 Java 的这一套 " 语义统一 " 对设计原则的支持

- **Abstraction**：不用在脑子里同时维护 " 值对象/指针对象/栈/堆/析构 "
- **Least Surprise（部分）**：所有对象变量行为一致（都是引用），参数传递规则一致（都是按值）
- **Testability**：如果配合不可变对象和小副作用设计，更容易写单元测试

> [!tldr] 你应该带走的 4 条铁律
> 1. **Java 只有按值传递**（pass-by-value）
> 2. Object 变量的 " 值 " 是引用，所以传的是**引用的副本**
> 3. `a = new …` / `a = 100` 这种是 " 改绑定 "，**改不到外面**
> 4. `a.field = …` / `a.add(…)` 这种是 " 改对象内部 "，**外面能看到**（前提对象可变）

# 3 Java 标准库 vs C++ STL

## 3.1 核心容器对照（最常用、最重要）

|C++ STL|Java 标准库|说明|
|---|---|---|
|`std::vector<T>`|`java.util.ArrayList<T>`|**最常用**的顺序容器|
|`std::list<T>`|`java.util.LinkedList<T>`|双向链表|
|`std::map<K,V>`|`HashMap<K,V>` / `TreeMap<K,V>`|哈希 vs 红黑树|
|`std::set<T>`|`HashSet<T>` / `TreeSet<T>`|同上|
|`std::iterator<T>`|`Iterator<T>`|用法相似|

### 3.1.1 关键理解点（比记名字重要）

1️⃣ **Java 有两套 map / set**

- `HashMap / HashSet`
	- 哈希表
	- 无序
	- 最常用、最快
- `TreeMap / TreeSet`
	- 平衡树（红黑树）
	- 有序
	- 语义上**最接近 C++ 的 map/set**

📌 讲义特别注明：

> C++ 的 `map/set` 是平衡树 → **语义上更像 TreeMap / TreeSet**，
> 但工程实践中 **HashMap / HashSet 更常见**。

## 3.2 字符串：`std::string` vs `java.lang.String`

|C++|Java|
|---|---|
|`std::string`|`String`|

但**语义差别巨大**：

- Java `String`：
	- **不可变（immutable）**
	- 拼接会创建新对象
- C++ `std::string`：
	- 可变
	- 原地修改

👉 设计含义：

- Java 把 " 字符串可变性 " 问题直接封死
- 换来：
	- 线程安全
	- 更少副作用
	- 更好的可测试性

（这和你刚学的 *Object vs Primitive* 是一条主线）

## 3.3 IO：这是差异最大的部分（不用背，用时查）

|C++|Java|
|---|---|
|`std::istream`|`InputStream` / `Reader`|
|`std::ostream`|`OutputStream` / `Writer`|
|`std::ifstream`|`FileReader`（常配 `BufferedReader`）|
|`std::ofstream`|`PrintWriter`|
|`std::stringstream`|`ByteArrayInputStream / OutputStream`|

### 3.3.1 关键思想差异（只记这一条）

> **Java IO 是 " 层层包装（Decorator）"，
> C++ IO 是 " 类层次 + operator 重载 "。**

你之后看到：

```java
BufferedReader br =
  new BufferedReader(new FileReader("x.txt"));
```

不要慌——这是 Java IO 的**标准姿势**。

## 3.4 `Vector` 的特殊提醒（很容易被误用）

讲义提到：

> `java.util.Vector<T>` is just like ArrayList, except thread-safe

含义是：

- `Vector` 是**老 API**
- 所有方法都同步（synchronized）
- 结果：
	- 慢
	- 粒度粗
- **现代 Java 几乎不用它**

📌 工程结论：

👉 **默认用 `ArrayList`，不要用 `Vector`**

> [!tldr] 这张表真正想教你的是什么？
> 不是 API 名字，而是 **迁移思维**：
>
> 1️⃣ **别试图 " 按 C++ 的方式写 Java"**
>
> - Java 没有运算符重载
> 
> - Java 容器行为更受接口与契约约束
> 
> 
> 2️⃣ **优先选语义，再看性能**
>
> - 要有序？`TreeMap`
> 
> - 要快？`HashMap`
> 
> - 要线程安全？后面再用并发容器（不是 `Vector`）
> 
> 
> 3️⃣ **Java 标准库是 " 设计导向 " 的**
>
> - 不可变对象
> 
> - 明确区分接口 vs 实现
> 
> - 强调组合、包装、契约

# 4 Java IO 的设计哲学

## 4.1 Java IO 的核心思想：Wrapper / Decorator

> **Java IO 的设计核心不是 " 一个类干所有事 "，
> 而是：每个类只干一件事，通过 " 包装 " 把能力叠加起来。**

这就是经典的 **Decorator Pattern（装饰器模式）**。

## 4.2 Java IO 的两条 " 正交维度 "

Java 在 IO 设计中，把 " 问题 " 拆成了两条**完全独立的维度**：

### 4.2.1 字节 vs 字符（What is being processed?）

#### 4.2.1.1 InputStream / OutputStream—— **字节**

- 原始数据（binary）
- 适合：
	- 文件
	- 网络
	- 图片
	- 协议
- 不关心编码

#### 4.2.1.2 Reader / Writer—— **字符**

- Unicode 字符
- 适合：
	- 文本
	- 行
	- 字符串
- 需要编码（UTF-8 / ASCII / …）

👉 **这是 Java 非常 " 工程化 " 的一刀切**：

> *" 你现在是在处理 bytes，还是 characters？"*
> 必须明确。

### 4.2.2 功能分层（What functionality is added?）

每一层 wrapper **只增加一个能力**：

|类|增加的能力|
|---|---|
|`FileReader`|从文件读字符|
|`InputStreamReader`|把 byte → char|
|`BufferedReader`|提供缓冲 + `readLine()`|
|`PrintWriter`|方便写文本|

## 4.3 经典示例 1：从文件读 " 行 "

讲义 Page 28 的例子是 **教科书级的 Decorator 示范**：

```java
FileReader reader = new FileReader("myFile.txt");
BufferedReader br = new BufferedReader(reader);
```

## 4.4 每一层干什么？

```
文件中的字节
   ↓
FileReader        // 读字符
   ↓
BufferedReader    // 提供 readLine()
```

📌 设计亮点：

- `FileReader` 不需要知道 " 什么是行 "
- `BufferedReader` 不关心 " 数据来自哪里 "
- 两者 **解耦**

## 4.5 经典示例 2：从 `System.in` 读 " 行 "

这是很多人第一次觉得 Java IO" 怪 " 的地方，但恰恰是它最优雅的地方。

```java
InputStreamReader isr = new InputStreamReader(System.in);
BufferedReader br = new BufferedReader(isr);
```

## 4.6 分层解释（非常重要）

```
System.in           // InputStream：字节
   ↓
InputStreamReader   // byte → char（编码）
   ↓
BufferedReader      // char → 行
```

你可以发现：

- `BufferedReader` **完全不知道**数据是来自文件还是键盘
- 它只知道：" 我下面有一个 Reader"

👉 这就是 **依赖抽象，不依赖具体实现**（DIP 的早期体现）

## 4.7 为什么 Java 要这样设计？（对比 C++）

### 4.7.1 C++ IO 的特点

- 强依赖继承层次
- 强依赖运算符重载（`<<`, `>>`）
- 一个类往往 " 功能很多 "

### 4.7.2 Java 的选择

- ❌ 不靠 operator overloading
- ✅ 靠组合 + 接口
- 每个类 **单一职责**

👉 非常符合：

- SRP（单一职责）
- OCP（可以加 wrapper，而不用改原类）
- DIP（高层只依赖 Reader / Writer 接口）

## 4.8 工程级大坑：❌ 不要 wrap 同一个流两次

这是 **Page 30 明确点名的坑**，而且是**真实世界会炸的那种 bug**。

### 4.8.1 错误示例（讲义原意）

```java
InputStreamReader isr = new InputStreamReader(System.in);
BufferedReader br = new BufferedReader(isr);

// 后面某处（千万别这样）
InputStreamReader isr2 = new InputStreamReader(System.in);
BufferedReader br2 = new BufferedReader(isr2);
```

### 4.8.2 为什么这是灾难？

#### 4.8.2.1 核心原因

- **wrapper 可能会 " 多读 " 数据做缓冲**
- 第一次 wrap 已经把一部分字节读进缓冲区
- 第二次 wrap：
	- 看不到那些字节
	- 或者卡住
	- 或者顺序错乱

这类 bug：

- 不报错
- 随机发生
- 极难调试

👉 这是**所有 IO 系统中最阴险的一类问题**

### 4.8.3 正确工程做法（讲义强调）

> ✅ **Wrap once, reuse everywhere**

```java
BufferedReader br =
  new BufferedReader(new InputStreamReader(System.in));

// 后面所有地方都用 br
```

这不是 " 风格建议 "，而是：

> **IO 系统的正确性前提**

# 5 对象序列化

## 5.1 什么是 Java 对象序列化？

### 5.1.1 `Serializable` 接口

```java
class Point implements Serializable { … }
```

- **标记接口（marker interface）**
- 不要求你实现任何方法
- 含义只有一句话：

	> " 这个对象**允许**被序列化 "

👉 这是 Java 早期设计中很典型的风格。

### 5.1.2 `ObjectOutputStream` / `ObjectInputStream`

- `ObjectOutputStream` 把 **对象图（object graph）** 写到输出流
- `ObjectInputStream` 从输入流中 **还原对象图**

```java
ObjectOutputStream oos =
  new ObjectOutputStream(new FileOutputStream("data.bin"));
oos.writeObject(obj);

ObjectInputStream ois =
  new ObjectInputStream(new FileInputStream("data.bin"));
MyType x = (MyType) ois.readObject();
```

📌 注意关键词：**对象图**

- 不只是一个对象
- 它引用的对象，也会被递归写出

## 5.2 致命但 " 非常真实 " 的工程陷阱

> 写同一个对象两次 ≠ 写两个快照

这是 Page 32 特别强调、而且**非常反直觉**的一点。

### 5.2.1 看似合理的代码

```java
oos.writeObject(obj);   // obj 状态是 A
obj.setX(42);
oos.writeObject(obj);   // 你以为写的是状态 B
```

### 5.2.2 实际发生了什么？

- `ObjectOutputStream` **会记住**：

> " 这个对象我已经写过了 "

- 第二次写：
	- 不会再写对象内容
	- 只写一个 " 引用标记 "
- 结果：
	- 读回来时，两次得到的对象
	- **状态是第一次的版本**

💥 典型后果：

> " 我明明改了对象，为什么读回来还是旧数据？"

### 5.2.3 这是 bug 吗？

不是。这是 **Java 序列化机制的 " 特性 "**。

设计初衷：

- 正确处理对象图中的**共享引用**
- 防止重复序列化、无限递归

但代价是：

- **极其不直观**
- 极其容易误用

### 5.2.4 唯一补救手段（讲义提到）

`oos.reset();`

- 强制让 stream " 忘掉 " 之前写过的对象
- 之后再写，会当作新对象

📌 但注意：

> 如果你已经需要 `reset()`，
> 那你大概率已经在用错这个机制了。

## 5.3 为什么这在真实工程中很危险？

### 5.3.1 格式不可读（Not human-readable）

- 二进制
- 无法直接打开看
- Debug 极其痛苦

### 5.3.2 行为 " 隐式 + 状态相关 "

- 是否写新数据，取决于之前有没有写过这个对象
- 很容易出现：
	- 时间相关 bug
	- 顺序相关 bug

### 5.3.3 对演化极不友好

- 类结构一变：
	- 加字段
	- 改类型
- 老数据可能直接读不回来（`InvalidClassException`）

## 5.4 所以：为什么很多工程更偏向 JSON / 明确格式？

讲义在 Page 32 点了一句，但背后是**非常成熟的工程共识**。

### 5.4.1 JSON / 明确格式的优势

#### 5.4.1.1 显式快照语义

- 每次序列化就是 " 当前状态的完整表示 "
- 没有 " 你之前写过没写过 " 的隐式行为

#### 5.4.1.2 可读、可调试

- 出问题时：
	- 直接打开文件
	- 就能看到数据

#### 5.4.1.3 跨语言 / 跨系统

- Java ↔ Python ↔ Go
- 天然支持

#### 5.4.1.4 更适合版本演进

- 字段可以：
	- 增
	- 删
	- 给默认值
- 向后兼容更容易

# 6 Java vs C++：语言差异如何影响设计原则

## 6.1 方法分派（Dynamic Dispatch）

### 6.1.1 C++：你必须 " 显式请求 " 动态分派

在 C++ 中：

```cpp
class Base {
public:
  virtual void f();
};
```

如果你忘了 `virtual`：

```cpp
Base* b = new Derived();
b->f();   // 调用的是 Base::f，而不是 Derived::f
```

**问题在哪里？**

- 子类明明 override 了方法
- 但调用时却 " 静态绑定 "
- 行为和直觉不一致

👉 这是一个**非常典型的违反 Least Surprise 的例子**。

### 6.1.2 Java：**所有实例方法默认都是动态分派**

在 Java 中：

```java
class Base {
  void f() { … }
}
class Derived extends Base {
  @Override
  void f() { … }
}

Base b = new Derived();
b.f();   // 一定调用 Derived.f()
```

**你没有任何 " 选择权 "**：

- 不能关掉动态分派（除非 `static` / `final` / `private`）
- 默认行为就是 " 多态 "

### 6.1.3 这如何支持设计原则？

#### 6.1.3.1 Least Surprise（最小惊讶原则）

- 你写了 override
- 那我就一定用 override 的版本
- 不会因为 " 忘了一个关键字 " 而 silently 出 bug

#### 6.1.3.2 Open/Closed Principle（开闭原则）

- C++：
	- 如果父类方法没声明 virtual
	- 你**必须修改父类**才能支持多态
- Java：
	- 父类天生支持被扩展
	- 子类只需 override

👉 Java 在语言层面**偏向 " 可扩展性 "**，哪怕牺牲一点性能控制。

## 6.2 内存管理 & RAII

这一部分是 **Page 37–43 的核心思想**，也是 C++ 程序员 " 最不适应 Java" 的地方。

### 6.2.1 C++：RAII（资源获取即初始化）

#### 6.2.1.1 RAII 的核心思想

```cpp
{
  std::lock_guard<std::mutex> lock(m);
  // 临界区
} // 作用域结束 → 自动解锁
```

- 对象：
	- 在栈上创建
	- 作用域结束 → 析构函数自动调用
- 析构函数：
	- 是资源释放的 " 最后防线 "
- 即使抛异常：
	- 析构函数仍然会执行

👉 **RAII = 自动清理 + 异常安全**

### 6.2.2 Java：彻底放弃 RAII

Java 的选择是**极端的**：

- ❌ 没有栈对象（所有对象都在 heap）
- ❌ 没有析构函数
- ✅ GC 自动回收 " 内存 "
- ❌ 但 GC **不管非内存资源**

于是问题来了（Page 39–40）：

> 如果没有 RAII，那这些怎么办？

- 文件要 close
- 锁要 unlock
- socket 要 release
- 如果中途抛异常怎么办？

### 6.2.3 这是一个 " 设计原则级别 " 的冲突

Java 的取舍是：

|放弃|换来|
|---|---|
|手动资源生命周期控制|更简单的内存模型|
|RAII|更少内存错误|
|确定性析构|GC + 抽象一致性|

👉 这非常符合 Java 的整体哲学：

> **" 宁可限制表达能力，也要减少灾难性错误。"**

## 6.3 Java 的解决方案：把 " 资源管理 " 变成语言级保证

Java 没有 RAII，但它并不是 " 不管资源 "。

它做了**三件非常关键的事**。

### 6.3.1 `synchronized`：锁是语言级的

在 Java 中：

```java
synchronized(obj) {
  // critical section
}
```

**关键特性（Page 44）**：

- 锁和对象绑定（每个对象都有内置 mutex）
- **无论正常退出还是异常退出**都会自动释放锁

👉 这在语义上**模拟了 RAII 的 " 异常安全 "**

对比 C：

```c
pthread_mutex_lock(&m);
// 如果这里抛异常或 return？
pthread_mutex_unlock(&m);
```

Java 把 "unlock 必须发生 " 变成了**语法保证**。

### 6.3.2 `try / catch / finally`：显式兜底

```java
try {
  open file
  do stuff
} catch (Exception e) {
  handle
} finally {
  close file
}
```

`finally` 的语义是：

> **" 不管发生什么，这段代码一定执行。"**

这相当于：

- 没有析构函数
- 但你可以写一个 " 手动析构区 "

### 6.3.3 `try-with-resources`：最接近 RAII 的 Java 方案

这是 Java 对 RAII 的**正式回应**（Page 52）：

```java
try (BufferedReader br = new BufferedReader(…)) {
  // 使用 br
}
// 自动 close
```

前提：

- 资源实现 `AutoCloseable`

效果：

- 作用域结束 → 自动关闭
- 异常安全
- 语法层面强制正确使用

👉 这是 Java 对 RAII 的**" 有限复刻版 "**

## 6.4 这三点如何一起服务设计原则？

我们把它们统一起来看：

### 6.4.1 Least Surprise

- 动态分派默认开启
- synchronized 自动释放
- try-with-resources 自动关闭

→ 行为**符合直觉**

### 6.4.2 Open/Closed Principle

- 子类自然可扩展
- 不需要修改父类声明
- 资源管理方式统一、可组合

### 6.4.3 Design for Testability

- 不依赖析构时机
- 资源释放路径显式
- 测试中更可控、更可复现

# 7 Java 异常系统

## 7.1 一、Java 的 " 强设计选择 "：Checked Exception

### 7.1.1 1️⃣ 什么是 Checked Exception？

Java 把异常分成两大类（Page 47–48）：

```java
Throwable
├── Error
└── Exception
    ├── RuntimeException   (unchecked)
    └── Other Exceptions   (checked)
```

### 7.1.2 Checked Exception 的定义

> **凡是 " 不是 RuntimeException 的 Exception"，
> 都必须在编译期被处理。**

" 被处理 " 只有两种方式：

1. **捕获（catch）**
2. **声明传播（throws）**

## 7.2 一个最直观的例子（Page 50）

```java
new FileInputStream(name);
```

这行代码 **可能失败**（文件不存在），因此它：

- 抛出 `FileNotFoundException`
- 这是一个 **Checked Exception**

### 7.2.1 如果你什么都不做

```java
public void f() {
  new FileInputStream(name); // ❌ 编译错误
}
```

### 7.2.2 你必须显式选择一种立场

#### 7.2.2.1 方案 A：我来处理

```java
try {
  new FileInputStream(name);
} catch (FileNotFoundException e) {
  // handle
}
```

#### 7.2.2.2 方案 B：我不处理，但我承认风险

```java
public void f() throws FileNotFoundException {
  new FileInputStream(name);
}
```

👉 **没有第三条路**。

## 7.3 为什么 Java 要这样 " 折磨 " 你？

这是一个**极其有意识的语言设计决定**。

### 7.3.1 Java 的立场是

> **" 可能失败的操作，不应该被悄悄忽略。"**

## 7.4 Checked Exception 的优点（为什么它服务设计原则）

### 7.4.1 Least Surprise（最小惊讶原则）

Java 的异常设计在做一件事：

> **把 " 可能失败 " 变成 API 的显式部分。**

当你看到：

```java
void readFile() throws IOException
```

你**提前知道**：

- 这行代码可能失败
- 失败类型是什么
- 调用者必须考虑

👉 不会出现：

- " 这玩意儿怎么突然抛异常了？"
- " 文档里根本没说会失败啊？"

### 7.4.2 强迫设计者思考失败路径（非常重要）

在 Java 中，设计 API 时你必须回答：

- 这个方法**会失败吗**？
- 如果会，失败是：
	- 可恢复的？
	- 逻辑错误？
	- 系统错误？
- 谁来负责处理？

因为你：

- **要么 catch**
- **要么 throws**

👉 **异常成了设计的一部分，而不是实现细节。**

这在软件工程里是一个非常强的信号：

> *" 失败不是边角情况，而是核心路径之一。"*

---

### 7.4.3 改善系统级鲁棒性

在大型系统中：

- IO
- 网络
- 数据库
- 外部服务

失败是**常态**，不是异常。

Checked Exception 迫使：

- 底层明确 " 我可能失败 "
- 上层明确 " 我怎么处理失败 "

---

## 7.5 为什么 Java 还要区分 RuntimeException？

### 7.5.1 RuntimeException（Unchecked）

例如：

- `NullPointerException`
- `ArrayIndexOutOfBoundsException`

特点：

- **几乎任何地方都可能发生**
- 往往代表：
	- 程序 bug
	- 逻辑错误
- 通常：
	- 不可恢复
	- 不值得在 API 层层声明

👉 Java 的态度是：

> **" 程序员错误，不需要污染方法签名。"**

## 7.6 try / catch / finally：为 Checked Exception 服务的语法

### 7.6.1 `finally`（Page 46）

```java
try {
  // 可能失败
} catch (Exception e) {
  // 处理失败
} finally {
  // 一定会执行
}
```

`finally` 的语义是：

> **" 不管你怎么离开 try，这里一定执行。"**

这是 Java 在 **没有析构函数** 的前提下，为资源安全兜底的重要机制。

### 7.6.2 try-with-resources（Page 52）

```java
try (BufferedReader br = new BufferedReader(…)) {
  // 使用 br
}
// 自动 close
```

- 编译器生成隐式 `finally`
- 异常安全
- 极大减少样板代码

👉 **这是 Checked Exception + 资源管理 的最佳组合**

## 7.7 缺点：为什么很多人 " 讨厌 " Checked Exception？

讲义点到为止，但你应该现在就能理解原因。

### 7.7.1 啰嗦（Verbosity）

```java
try {
  …
} catch (IOException e) {
  throw new RuntimeException(e);
}
```

很多时候：

- 你 " 没法真正处理 "
- 只是为了让编译器满意

### 7.7.2 污染接口（Exception Leakage）

- 一个底层异常
- 会被迫一路 `throws`
- 传到最顶层

API 看起来像：

```java
f() throws A, B, C, D, E
```

### 7.7.3 在函数式 / 高阶 API 中很难用

- Lambda
- Streams
- 回调

Checked Exception 往往**非常不搭**

👉 这也是为什么：

- Java 8 之后
- 社区对 Checked Exception 的态度更加保守

# 8 Java 继承与接口

## 8.1 Java 的根本选择：单继承

### 8.1.1 Java：**只能继承一个类**

```java
class Bat extends Animal { … }   // OK
class Bat extends Animal, Mammal { // ❌ 不允许
}
```

对比：

- C++：支持多继承
- Java：**明确禁止多继承（类）**

### 8.1.2 为什么 Java 要这么 " 狠 "？

因为多继承在工程上会带来：

- **菱形继承问题（Diamond Problem）**
- 方法冲突
- 状态重复
- 难以预测的初始化/析构顺序

Java 的设计立场是：

> **宁可限制表达能力，也不要引入 " 你很难证明正确性 " 的复杂性。**

## 8.2 多重能力怎么办？——接口（Interface）

Java 的答案不是 " 少表达 "，而是**换一种表达方式**。

### 8.2.1 接口：表达 " 能做什么 "，而不是 " 是什么 "

```java
public interface FlyingCreature {
  void flyTo(Location where);
}
```

接口的核心特征（Page 55–56）：

- ❌ 没有实例字段（没有状态）
- ❌ 不负责实现（Java 8 前）
- ✅ 只描述 **行为契约（capability / role）**

类可以：

```java
class Bird extends Animal implements FlyingCreature { … }
class Bat  extends Animal implements FlyingCreature { … }
```

📌 这就是讲义里的图示含义：

- `Animal`：本体（是什么）
- `FlyingCreature` / `Quadruped`：能力（能做什么）

## 8.3 抽象类 vs 接口：各自的角色

讲义 Page 56 给了一个对照示例，非常典型。

### 8.3.1 抽象类（Abstract Class）

```java
public abstract class Animal {
  private String name;

  public Animal(String name) {
    this.name = name;
  }

  public String getName() { return name; }

  public abstract void eat(Food f);
}
```

**适合：**

- 有**共享状态**
- 有**部分实现**
- 强 is-a 关系

### 8.3.2 接口（Interface）

```java
public interface FlyingCreature {
  void flyTo(Location where);
}
```

**适合：**

- 纯行为契约
- 无状态
- 多重能力组合

## 8.4 接口如何直接支持 SOLID？

### 8.4.1 Interface Segregation Principle（接口隔离原则）

> **Clients should not be forced to depend on methods they do not use.**

#### 8.4.1.1 如果没有接口隔离，会发生什么？

假设你设计了一个 " 万能接口 "：

```java
interface Animal {
  void eat();
  void fly();
  void swim();
  void run();
}
```

问题：

- 狗不会飞
- 鱼不会跑
- 被迫实现一堆没意义的方法

👉 **这是典型的 ISP 违反。**

#### 8.4.1.2 Java 接口的自然解决方案

把能力拆开：

```java
interface Flyable { void fly(); }
interface Swimmable { void swim(); }
interface Runnable { void run(); }
```

类只实现**它真正需要的接口**。

📌 Java 的语言规则（多接口）**天然鼓励接口拆分**，而不是 " 上帝接口 "。

### 8.4.2 Dependency Inversion Principle（依赖反转原则）

> **High-level modules should not depend on low-level modules.
> Both should depend on abstractions.**

#### 8.4.2.1 Java 接口是 DIP 的 " 第一公民 "

对比两种设计：

##### 8.4.2.1.1 ❌ 依赖具体类

```java
class Zoo {
  private Bird bird;
}
```

#### 8.4.2.2 ✅ 依赖接口

```java
class Zoo {
  private FlyingCreature flyer;
}
```

结果：

- `Zoo` 不关心：
	- 是 Bird
	- 还是 Bat
	- 还是 Drone（以后加的）
- 只关心：**能不能飞**

👉 这正是 DIP 的精髓。

## 8.5 为什么 Java 的 " 接口 " 比 C++ 的 " 多继承 " 更适合工程？

### 8.5.1 关键差异不是 " 能不能 "，而是 " 引导你怎么做 "

|方面|C++ 多继承|Java 接口|
|---|---|---|
|状态|可继承多个|不允许|
|行为|混在一起|明确分离|
|冲突|常见|几乎无|
|设计倾向|强表达能力|强设计约束|

Java 用接口在**语言层面逼你回答一个问题**：

> **" 我是在建模 ' 是什么 '，还是在建模 ' 能做什么 '？"**

# 9 泛型 vs 模板

## 9.1 共同点：它们解决的是同一个问题

### 9.1.1 什么是参数化多态（Parametric Polymorphism）？

> **写一次代码，对 " 任意类型 T" 都成立。**

例如：

- `List<int>` / `List<String>`
- `LinkedList<double>` / `LinkedList<Point>`

目的：

- 代码复用
- 抽象算法
- 不为每种类型重写一份逻辑

这一点 **C++ 模板和 Java 泛型是完全一致的目标**。

## 9.2 C++：模板（Templates）

### 9.2.1 核心机制

- **编译期展开**
- 对每个 `T`：
	- 编译器生成一份 " 专用代码 "
- 类型检查发生在：
	- **使用点（instantiation point）**

### 9.2.2 直接后果

- 模板代码必须：
	- 完全可见（通常写在 `.h`）
- 错误往往：
	- 发生在 " 你第一次用它的时候 "
	- 报错信息巨大、晦涩

### 9.2.3 思维模型

> **" 模板不是代码，而是代码生成规则。"**

## 9.3 Java：泛型（Generics）

### 9.3.1 核心机制

- **编译一次**
- **类型擦除（Type Erasure）**
	- 编译后，`T` 会被擦成 `Object`（或上界）
- 类型检查发生在：
	- **定义点（definition point）**

### 9.3.2 直接后果

- 编译后的 `.class` 文件：
	- 对所有 `T` 复用
- 运行时：
	- **不知道 T 是什么**

### 9.3.3 思维模型

> **" 泛型是对类型能力的静态约束，而不是代码生成。"**

## 9.4 一个对比直觉（非常重要）

|维度|C++ 模板|Java 泛型|
|---|---|---|
|生成代码|每个 T 一份|只有一份|
|类型信息|运行期可见|运行期被擦除|
|错误发现|使用时|定义时|
|表达能力|极强|受限|
|设计倾向|最大灵活性|最大可预测性|

👉 **这不是技术优劣，而是设计取舍。**

## 9.5 为什么 Java 要选 " 类型擦除 "？

这是一个**设计原则问题**，不是历史偶然。

### 9.5.1 向后兼容（工程现实）

- Java 在引入泛型时：已有大量旧代码、JVM、库
- 类型擦除：
	- 不需要改 JVM
	- 老代码能继续跑

### 9.5.2 抽象优先，而不是 " 黑魔法 "

Java 的立场是：

> **" 如果一个泛型类型在逻辑上不适用于某些 T，那就不该让它编译通过。"**

这直接影响了 **Abstraction** 和 **LSP**。

## 9.6 泛型如何支持 Abstraction？

### 9.6.1 Java 强迫你明确 " 我需要 T 具备什么能力 "

看讲义 Page 59 的问题：

> 想排序？不是所有类型都能比较
> 想判等？不是所有类型都有合理相等关系

### 9.6.2 Java 的解决方案：**有界泛型（Bounded Type Parameters）**

```java
class Glass<T extends Liquid> { … }
```

含义是：

- 我不是 " 对任何 T 都成立 "
- 我只对：
	- `T` 是 `Liquid` 子类
	- 或实现了某接口
		的情况成立

👉 **抽象变得明确、可解释、可证明**

### 9.6.3 对比 C++ 模板

```java
template<typename T>
void sort(T a[], int n);
```

- 你**假设**：
	- `T` 支持 `<`
- 但语言不强制说明
- 错误只会在 " 第一次用到不支持 `<` 的 T" 时爆炸

👉 Java 的设计是：

> **" 不要让我猜你需要什么能力，写出来。"**

## 9.7 泛型与 LSP（参数化版本）

讲义 Page 64 给了一个非常精彩的说法：

> **"Parametric analog of the Liskov Substitution Principle"**

我们把它翻成 " 人话 "。

### 9.7.1 经典 LSP（回顾）

> 如果 `S` 是 `T` 的子类型，
> 那么任何使用 `T` 的地方，都可以安全地使用 `S`。

### 9.7.2 参数化版本的 LSP

> 如果一段代码是 **对类型参数 `<T>` 成立的**，
> 那么任何满足约束的 `S`，
> 都应该能安全地替换 `T`。

### 9.7.3 Java 泛型是如何 " 逼你做到这一点 " 的？

- 你在定义时：明确写出 `T` 的上界 / 接口
- 编译器在定义点：检查你的代码是否 **对所有合法 T 都成立**

👉 换句话说：

> **Java 泛型在 " 写代码时 " 就验证了参数化 LSP。**

## 9.8 为什么说：C++ 模板更强，但 Java 泛型更 " 工程化 "？

### 9.8.1 C++ 模板的优点

- 极强表达能力
- 零运行时成本
- 可以写非常高性能、通用的库（STL）

### 9.8.2 Java 泛型的优点

- 错误更早暴露
- API 行为更稳定
- 使用者更少 " 意外惊喜 "

ECE 651 的立场很明确：

> **在软件工程语境下，" 可预测性 " 通常比 " 最大表达力 " 更重要。**

# 10 运算符重载 & 隐式转换

## 10.1 C++ 给了你什么？（以及为什么这很危险）

### 10.1.1 运算符重载（Operator Overloading）

C++ 允许你做这些事：

```cpp
Point a, b;
Point c = a + b;   // 看起来像数学
```

甚至：

```cpp
if (a < b) { … }
```

**问题不在于 " 能不能 "，而在于：**

- `+` 到底意味着什么？
- `a < b` 是按 x 比？y 比？距离比？

👉 含义**完全由作者决定**。

### 10.1.2 隐式转换（Implicit Conversion）

C++ 还允许：

- 单参数构造函数
- `operator type()`

```cpp
void f(Point p);
f(3);   // 可能通过隐式构造发生
```

**风险：**

- 编译通过
- 运行行为却不直观
- Bug 出现得非常隐蔽

## 10.2 Java 的态度：**直接不提供**

### 10.2.1 Java 明确选择

- ❌ **不允许用户自定义运算符重载**
- ❌ **不允许用户自定义隐式类型转换**

唯一的 " 例外 "：

- `+` 用于字符串拼接 👉 这是语言内建、语义固定的特例

## 10.3 Java 为什么这么 " 保守 "？

### 10.3.1 容易被滥用（讲义原话）

运算符：

- 看起来简单
- 实际语义隐藏在实现里

在大型工程中：

- 读代码的人**不知道你定义了什么语义**
- 必须跳转到定义才能确认

👉 **认知成本极高**

### 10.3.2 严重违反 Least Surprise（×100）

讲义的原话非常直接：

> **Least Surprise × 100**

含义是：

- 当我看到 `a + b`
- 我期待的是：
	- 数学加法
	- 或至少 " 显而易见 " 的含义
- 而不是：
	- 写数据库
	- 发网络请求
	- 修改全局状态

Java 的立场是：

> **如果语义不是显而易见的，就不要让它看起来 " 很简单 "。**

## 10.4 Java 给你的替代方案是什么？

### 10.4.1 明确的方法名

```java
Point c = a.add(b);
```

优点：

- 意图清晰
- 可搜索
- 可文档化
- 可测试

### 10.4.2 方法重载（参数列表重载）是允许的

```java
add(Point p)
add(int dx, int dy)
```

但：

- 名字相同
- 参数不同
- **语义仍然集中在 "add" 这个词上**

## 10.5 这是一个非常典型的 " 设计取舍案例 "

### 10.5.1 Java 放弃了什么？

- 表达力
- 写 " 像数学公式一样 " 的代码
- 一部分性能优化空间

### 10.5.2 Java 换来了什么？

- 代码可读性
- 行为可预测性
- 团队协作下的一致理解
- 更少 " 你以为是 A，其实是 B" 的 bug

# 11 单元测试（JUnit）

## 11.1 Java 的测试为什么是一等公民？

### 11.1.1 语言 + 生态共同 " 推着你测试 "

在 Java 世界里：

- 官方支持（JUnit 是事实标准）
- IDE / 编辑器深度集成
- 构建工具默认支持（Maven / Gradle）

👉 **不写测试反而是 " 逆流而行 "**。

## 11.2 目录结构明确（这是非常工程化的一点）

讲义直接给出了**标准项目结构**（Page 67）：

```java
src/
 ├── main/
 │    └── java/
 │         └── edu/duke/adh39/stuff/
 │              └── Something.java
 └── test/
      └── java/
           └── edu/duke/adh39/stuff/
                └── SomethingTest.java
```

### 11.2.1 这不是 " 约定俗成 "，而是**工具链假设**

- 构建工具默认：
	- 编译 `main` → 产物
	- 运行 `test` → 验证
- 测试类与被测类：
	- **包名一致**
	- 路径镜像

👉 这极大降低了 " 写测试 " 的心理与结构成本。

## 11.3 JUnit 的核心使用方式（不讲 API，只讲模式）

讲义里的示例非常经典：

```java
class SomethingTest {
  @Test
  public void test_someFun() {
    Something s = new Something(42);
    assertEquals(12, s.someFun(3));
    assertEquals(19, s.someFun(7));
  }

  @Test
  public void test_anotherFun() {
    Something s = new Something(99);
    assertEquals("hello", s.anotherFun());
  }
}
```

### 11.3.1 这里隐含了 3 个关键工程理念

1️⃣ **测试的是行为，不是实现**

2️⃣ **一个测试方法 = 一个小关注点**

3️⃣ **测试名本身就是文档**

## 11.4 强调的四个工程实践（Page 68）

这一页几乎可以直接当成**软件工程宣言**。

## 11.5 小步开发（Small Steps）

> **Write only a LITTLE bit of code before testing**

含义是：

- 不要一次写完一个类
- 写一个方法
- 甚至只写方法的一部分
- 然后立刻测试

👉 这本质上就是 **TDD / 迭代式开发** 的精神。

## 11.6 高频测试（Test Frequently）

> **Test frequently!**

- 每次小改动就跑测试
- 不等 " 功能全部完成 "

结果是：

- 错误定位更快
- 回滚成本极低

## 11.7 覆盖率（Coverage）

讲义强调：

> **Get 100% coverage**

这里的 "100%"**不是 KPI**，而是：

- 一个心理目标
- 一个信号：
	- " 我有没有漏掉重要分支？"

并且讲义明确提到：

- 行覆盖
- 分支覆盖（黄色 vs 绿色）

## 11.8 回归测试（Regression Testing）

这是整节的**终极目标**：

> **Refactor code → run tests → pass tests → be sure you did not break anything**

也就是说：

- 测试不是 " 找 bug 用的 "
- 测试是：**你敢不敢改代码的底气**

## 11.9 为什么 Java 特别适合 " 测试驱动的工程节奏 "？

把前面几讲串起来你会发现：

### 11.9.1 语言层面的支持

- 无指针算术
- 无手动内存释放
- 行为更可预测

### 11.9.2 设计层面的支持

- 接口 → 易 mock
- 不可变对象 → 易断言
- Checked Exception → 失败路径显式

👉 **测试在 Java 里成本更低、收益更高。**

## 11.10 为后续课程铺的 " 路 "

ECE 651 在这里埋了非常清晰的伏笔：

### 11.10.1 Testing（后面章节）

- 黑盒 / 白盒
- 单元 / 集成 / 系统
- Mock / Stub / Spy

### 11.10.2 Refactoring

- 重构 ≠ 重写
- 有测试，才能安全重构

### 11.10.3 CI（持续集成）

- 测试自动化
- 每次提交自动验证
- 不靠 " 人肉回归 "
