# 测试基本哲学

📍Page 2–7 

这一部分非常短，但它其实奠定了整门 " 软件工程测试观 " 的哲学基础。如果这一部分理解透彻，后面的所有测试技术都会变得 " 有方向 "。

## 什么是一个 " 好的 test case"？

讲义直接给出一句话：

> **能找到 bug 的测试才是好测试。** 

听起来简单，但背后其实有非常深刻的思想。

### 关键思想 1：测试的价值 = 找出错误

很多初学者会认为：

- 覆盖率高是好测试
- 运行成功是好测试
- 写得优雅是好测试

但本质上：

> 一个测试只有在发现 bug 时才真正产生价值。

换句话说：

- 如果测试永远都通过
- 永远发现不了问题

那它其实可能只是 " 摆设 "。

### 举个例子

假设我们写了一个排序函数：

```java
int[] sort(int[] arr)
```

你写了一个测试：

```java
assertEquals([1,2,3], sort([1,2,3]));
```

这个测试：

- 会通过
- 看起来没问题

但它可能毫无价值，因为：

- 它没有测试 " 乱序 "
- 没测试 " 重复 "
- 没测试 " 边界 "

真正好的测试是：

```java
assertEquals([1,2,3], sort([3,2,1]));
```

它才有机会发现 bug。

## 通过所有测试 ≠ 没有 bug

讲义引用了 Edsger Dijkstra 的一句经典话：

> **Testing can only show the presence of errors, not their absence.** 

翻译：

> 测试只能证明错误存在，不能证明错误不存在。

### 关键思想 2：测试是 " 负向证明 "

数学证明是：

> 如果成立 → 证明它永远成立

测试不是数学证明。测试的本质是：

> 在有限样本上进行尝试

但程序的输入空间通常是：

- 无限
- 或极其庞大

例如：

```java
int f(int x)
```

x 的取值范围是 2^32 种。你不可能全部测试。

### 举个极端例子

假设：

```java
if (x == 1234567) {  
    crash();  
}
```

你的测试从未测到 1234567。你会误以为程序是 " 正确 " 的。

所以：

✔ 测试通过 ≠ 程序正确

✔ 测试失败 ⇒ 一定有 bug

这是一种单向逻辑。

## 测试的目标是什么？

讲义强调：

> 测试的目标是找 bug，而不是证明正确。

这是和 " 学生心态 " 完全不同的地方。很多人写测试是为了：

- 让测试通过
- 满足覆盖率
- 通过 CI

但正确思维应该是：

> 我应该站在 " 破坏者 " 的角度。

测试工程师的角色是：🧨 想办法把程序炸掉。

## 两大类测试思想

讲义在 Page 6–7 提出两大类测试 

### 1️. Black Box Testing

特点：

- 不看代码
- 只看规格（specification）
- 从 " 用户角度 " 出发

问的问题是：

> 这个程序是否符合需求？

### 2️. White Box Testing

特点：

- 看代码
- 根据实现设计测试
- 关注代码路径

问的问题是：

> 哪些代码没被执行？
> 哪些分支没覆盖？

# 黑盒测试

📍Page 8–32 

这一部分是整个 Testing 体系的第一块 " 实战方法论 "。

如果说 Testing Review 是哲学层面，那 Black Box Testing 是第一种真正可落地的测试设计方法。

## 为什么用 Black Box？

📍Page 8–9 

讲义给出两个关键理由：

### 1️. 可以在写代码之前写测试（TDD）

Black Box 只依赖规格（specification）。

因此：

- 代码还没写
- 你已经可以设计测试

这就是：

> Test Driven Development (测试驱动开发)

### 2. 避免 " 实现偏见 "

如果开发者自己写测试：

- 很容易按照自己的思路写
- 很容易遗漏自己没想到的情况

黑盒测试强调：

> 从用户角度出发

它问的是：

- 输入是什么？
- 输出应该是什么？
- 需求可能被误解在哪里？

而不是：

- 我写了什么代码？

> [!NOTE] Black Box 的核心思想
> 一句话总结：
>
> > 根据规格划分输入空间，然后系统性覆盖它。
>
> 这和 White Box 完全不同。White Box 看代码路径；Black Box 看输入空间。

## 核心方法一：等价类划分（Equivalence Partitioning）

📍Page 12–15 

### 1. 什么是等价类？

把输入空间划分为若干 " 行为相同 " 的类别。假设：

> 如果某个输入出错，那么同类的输入都会出错。

于是：我们可以在每个类别中选一个代表测试。

### 2️. 类别类型

通常分为：

- ✅ Valid classes（有效类）
- ❌ Error classes（错误类）

### 3️. 经典例子：Triangle Problem

📍Page 14 

输入：a, b, c

分类：

- 等边
- 等腰
- 一般
- 非三角形

这就是四个等价类。你不需要测试所有可能的 (a,b,c)；只需要每类选代表。

### 4️. 等价类的本质

它是一个 " 降维 " 方法：无限输入 → 有限类别

否则你根本无法测试。

### 5️. 等价类的局限

📍Page 15 

理想情况：

> 一个类中一个坏，全部坏。

现实：

- 不一定
- 边界可能很特殊

这就引出了下一个方法。

## 核心方法二：边界值分析（Boundary Value Analysis, BVA）

📍Page 16 

### 1️. 为什么边界重要？

经验法则：

> 大多数 bug 出现在边界。

尤其：

- off-by-one
- > = vs >
- <= vs <

### 2️. 怎么做 BVA？

对于区间：

```
[0,45]
```

测试：

```
- -1
- 0
- 45
- 46
```

📍Page 16 明确举例 

![[Pasted image 20260304103519.png|300]]

### 3️. 不只是整数

例如字符串：

"XXX" 是否 prefix of "abde"？

长度边界也属于 BVA。

### 4️. 等价类 + BVA 的关系

等价类解决：

- 输入空间划分

BVA 解决：

- 类别之间的 " 边界问题 "

两者是组合使用的。

## 核心方法三：类别覆盖（Category Coverage）

📍Page 11–12 

讲义强调：你要系统覆盖输入类型。例如：

1. **对 List：**

- 空
- 一个元素
- 多个元素

2. **对 Integer：**

- 0
- 小正数
- 大正数
- 小负数
- 大负数

3. **对 Boolean：**

- true
- false

这个思想非常重要：

> 不要只测试 " 典型值 "，要测试 " 类别 "。

这是一种结构化测试思维。

## 一个完整黑盒设计流程

假设函数：

```java
boolean isValidAge(int age)
```

规则：age 必须在 0–150 之间

### Step 1：等价类

```
- age < 0 ❌
- 0 ≤ age ≤ 150 ✅
- age > 150 ❌
```

### Step 2：BVA

测试：

```
- -1
- 0
- 150
- 151
```

### Step 3：类别覆盖

考虑：

```
- 0
- 1
- 149
- 150
- 极端负值
- 极端正值
```

这就是完整 Black Box 设计。

## Black Box Testing 的优缺点总结

📍Page 32 

### 优点

✔ 从用户角度出发

✔ 不依赖实现

✔ 可在编码前设计

✔ 可由独立测试人员设计

✔ 有助于发现规格问题

### 缺点

❌ 无法保证所有路径被测试

❌ 依赖规格清晰

❌ 难以设计复杂系统测试

> [!NOTE] Black Box 的真正意义
> Black Box 是：
>
> > 从 " 需求空间 " 出发构造测试。
>
> White Box 是：
>
> > 从 " 实现空间 " 出发构造测试。
>
> Black Box 的本质是：
>
> > 测试规格是否被满足。

# 白盒测试

📍Page 33–52 

如果 Black Box 是 " 从规格出发 "，那么 White Box 是：

> 从代码结构出发设计测试。

> [!NOTE] White Box Testing 的核心思想
> **Black Box 问：**
>
> > 输入空间覆盖了吗？
>
> **White Box 问：**
>
> > 代码结构覆盖了吗？
>
> 它关注：
>
> - 哪些代码行被执行？
> 
> - 哪些分支被走到？
> 
> - 哪些路径组合没被测试？
> 
> 
> 这就引出了：**Code Coverage（三种覆盖率）**

## 1️. Statement Coverage（语句覆盖）

### 定义

每一行代码至少执行一次。

### 示例（讲义代码）

```java
int fun(int x) {  
    int num = 0;  
    if (x > 0) {  
        num = 1;  
    }  
    return x / num;  
}
```

📍Page 34 

如果测试：

```java
fun(5)
```

执行流程：

```
- int num = 0;
- if (x > 0) → true
- num = 1;
- return x / num;
```

所有语句都执行了吗？是的。

✔ 100% Statement Coverage

**但问题是你没有测试：**

```java
fun(-1)
```

会发生：

- num = 0
- if false
- return x / 0 → crash

所以：

> Statement Coverage ≠ 没有 bug

## 2️. Branch Coverage（分支覆盖）

### 定义

每个条件分支的 true / false 都被执行。

对于：

```java
if (x > 0)
```

你必须测试：

- x > 0
- x ≤ 0

### Branch 覆盖更强

Statement 只保证 " 行被执行 "，Branch 保证 " 每个条件方向都被执行 "。

📍Page 42 图示：

- 节点 = statement
- 边 = branch 

### 重要概念 1: Statement ≠ Branch

你可以有 100% statement coverage，但没有 branch coverage。例如：

```java
if (x > 0)
```

只测试正数：

- 所有语句执行
- 但 false 分支没测

## 3. Path Coverage（路径覆盖）

### 定义

覆盖所有可能的路径组合。

例如：

```java
if (a > 0)
	…

if (b > 0)
	…
```

路径：

- TT
- TF
- FT
- FF

共 4 条路径。

![[Pasted image 20260304141622.png|379]]

如果有 n 个 if：路径数 = 2\^n ⚠ 指数爆炸。

📍Page 35 提到 exponential number of tests 

### 重要概念 2: Path 可能指数级爆炸

真实系统：

- 10 个分支 → 1024 条路径
- 20 个分支 → 1,048,576 条路径

几乎不可能完全覆盖。

## 覆盖率层级关系

可以理解为：

```
Path Coverage

   ⬇

Branch Coverage

   ⬇

Statement Coverage
```

强度递增。

### 重要概念 3: Branch ≠ Correctness

📍Page 49–52 

讲义用 factorial 举例说明。假设实现：

```java
int fact(int n) {

	if (n == 0)

		return 1;

	return n * fact(n - 1);

}
```

测试：

```
fact(0) = 1
fact(1) = 1
fact(3) = 6
```

✔ Branch coverage achieved 

但假设 bug：

```java
return n + fact(n - 1); // 错误
```

某些测试仍然可能通过。所以：

> 即使 Branch Coverage 100%，仍然可能存在 bug。

### 覆盖率的真正意义

Coverage 只能告诉你：

- 哪些代码 " 没被执行 "

它不能告诉你：

- 执行了是否正确

它衡量的是：

> 测试的 " 广度 "，而非 " 正确性 "。

## 必要 vs 充分

📍Page 46 

### 哪种覆盖率是必要的？

讲义明确：✔ 至少需要 Statement Coverage。因为：

如果有代码从未执行，你根本不知道它是否有 bug。

### 通常需要 Branch Coverage

在 ECE651 作业中：Branch coverage 是满分要求

因为它比 statement 更严格。

### 但没有任何覆盖率是充分条件

即使：

- Path Coverage 100%

仍然不能证明程序正确。原因：

- 覆盖只检查 " 走没走 "
- 不检查 " 对不对 "

> [!NOTE] White Box Testing 的本质
> White Box 关注：**结构完整性**
>
> Black Box 关注：**规格完整性**
>
> White Box 是从：" 实现空间 " 找漏洞。

# 变异测试

📍Page 56–67 

如果说：

- Black Box → 检查 **规格覆盖**
- White Box → 检查 **代码覆盖**

那么 Mutation Testing 的目标是：

> **检查测试本身是否足够强。**

换句话说：

> **Testing the tests.**

## 为什么需要 Mutation Testing？

在前面的 White Box 部分我们已经看到一个问题。即使：

- ✔ Statement coverage 100%
- ✔ Branch coverage 100%

仍然可能：

❗ 测试没有发现 bug。

讲义在 Page 56 提出一个问题：

> How can we test that our test cases are good? 

也就是说：

> **我们如何验证测试集本身的质量？**

### Mutation Testing 的核心思想

讲义提出一个非常直观的思路：

📍Page 57 

> 如果我们写一些 " 稍微有 bug 的程序 "，
> 看测试是否能检测出来？

如果测试检测不出来，说明：

> 测试不够强。

## 什么是 Mutant（变异体）

Mutation Testing 的做法是：对程序做 **小修改**。生成一个新的程序版本：

> **mutant**

### 举例

原代码：

```java
if (x < 10)
```

Mutant：

```java
if (x <= 10)
```

或者原代码：

```java
i = i + 1;
```

Mutant：

```java
i = i + 2;
```

或者原代码：

```java
if (x == 0)
```

Mutant：

```java
if (x == 1)
```

讲义 Page 61 举例：

- `<` → `<=`
- `+1` → `+2`
- `0` → `1` 

## Mutation Testing 的流程

完整流程如下：

### Step 1

原程序：

```
Program P
```

### Step 2

生成很多 mutant：

```
P1  
P2  
P3  
…
```

每个 mutant **都有一个小 bug**。

### Step 3

运行测试集：

```
Test Suite T
```

### Step 4

观察结果：

- 如果测试失败 → mutant 被 **killed**
- 如果测试通过 → mutant **survives**

## Mutation Score

Mutation Testing 有一个指标：

**Mutation Score = killed mutants / total mutants**

例如：

```
90 / 100 = 90%
```

说明：测试可以发现 90% 的错误。

## 为什么 Mutation Testing 很强？

因为它在问一个更深的问题：

> 如果程序真的有 bug，我的测试能发现吗？

这比：

- coverage
- branch testing

更接近真实世界。

### 讲义中的 factorial 示例

📍Page 59 

讲义说明：

```java
fact(0) = 1  
fact(1) = 1  
fact(3) = 6
```

但存在一个错误实现：wrong but passes all test cases

也就是说：

✔ 测试全部通过

❌ 程序是错的

这说明：

> 测试不够强。

如果 mutation testing 我们会生成 mutant：

```java
n * fact(n-1)  
→  
n + fact(n-1)
```

如果测试不能发现：说明测试有问题。

## Mutation Testing 的局限

讲义 Page 62 提到两个问题 

### 1️. Mutant 的数量是无限的

理论上：

- 每一行代码
- 每个运算符

都可以变异。mutant 数量可能非常大。

### 2. Equivalent Mutants

有些 mutant 其实仍然是正确实现。

例如：

```java
if (x > 10)
```

改为：

```java
if (x >= 11)
```

逻辑完全一样。

这种 mutant：无论怎么测试都不会 fail。必须：

- 人工判断
- 排除

# Test Oracle 问题

📍Page 69 

讲义提出一个关键问题：

> **How do we judge if a program is correct or not?**

也就是：

> **我们如何知道测试结果是对还是错？**

## 1️. 什么是 Test Oracle

**Test Oracle**：

> 判断测试结果是否正确的标准。

换句话说：

```
输入 → 程序运行 → 输出  
                    ↓  
                  Oracle
```

Oracle 会判断：输出 == 正确结果?

### 举例

测试 factorial：

```
fact(3) → 6
```

Oracle：

```
expected = 6  
actual = fact(3)  
  
assert(expected == actual)
```

这里：

```
expected value = oracle
```

## 2. 什么时候 Oracle 很容易？

当问题是：

- 数学计算
- 排序
- 数据结构

例如：

```
sort([3,1,2]) → [1,2,3]
```

Oracle 很清楚。

## 3. 什么时候 Oracle 很难？

讲义给出两个现实例子：

📍Page 71–72 

### 例子 1：Google Translation

输入：

```
I like apples
```

输出：

```
我喜欢苹果
```

问题：有没有**唯一正确答案**？可能还有：

```
我爱吃苹果  
我很喜欢苹果
```

Oracle 很难定义。

### 例子 2：Database System

查询：

```sql
SELECT * FROM Students WHERE age > 20
```

如果数据库很大：

- 很难手算正确结果
- 很难写 expected output

## 4. Test Oracle Problem

于是出现问题：

> **没有 oracle，我们怎么测试？**

因为：没有正确答案 → 无法写 assert

# Metamorphic Testing（变形测试）

📍Page 70–73 

为了解决 Oracle problem，提出了：

> **Metamorphic Testing**

## Metamorphic Testing 的核心思想

核心思想：

> 我们不检查 " 单次结果是否正确 "，
> 而检查 " 多次执行之间是否满足某种关系 "。

换句话说，我们利用：

> **Metamorphic Relation (MR)**

## 什么是 Metamorphic Relation

讲义定义：

📍Page 70 

> Metamorphic relations are necessary properties of the intended functionality of software.

意思：

> 程序必须满足的一些数学或逻辑关系。

这些关系涉及：

> **多次运行的结果之间的关系**

而不是：

> 单次运行的正确结果。

# 五、经典例子：sin(x)

讲义例子：

📍Page 70 

数学关系：

```
sin(x + π) = sin(x)
```

测试方法：

```
y1 = sin(x)  
y2 = sin(x + π)  
```

```
assert(y1 == y2)
```

虽然我们不知道：

```
sin(2.31) = ?
```

但我们知道：

```
sin(2.31 + π) = sin(2.31)
```

## 几个经典 Metamorphic Relations

讲义给了几个例子。

### `add(a,b)`

📍Page 73 

加法满足：

```
add(a,b) = add(b,a)
```

测试：

```
add(2,5)  
add(5,2)  
```

**结果必须一样。**

还可以：

```
add(add(a,b),c) = add(a, add(b,c))
```

结合律。

### 2️. `sort(list)`

📍Page 73 

关系：

```
sort(shuffle(list)) = sort(list)
```

测试：

```
list = [3,1,2]  
shuffle → [1,3,2]  
  
sort(list) == sort(shuffled)
```

### 3️. `reverse(list)`

📍Page 73 

关系：

```
reverse(reverse(list)) = list
```

测试：

```
reverse(reverse([1,2,3])) == [1,2,3]
```

## Metamorphic Testing 的流程

完整流程：

**Step 1**

选一个输入

```
x
```

**Step 2**

运行程序

```
y = f(x)
```

**Step 3**

根据 MR 生成新输入

```
x'
```

**Step 4**

运行程序

```
y' = f(x')
```

**Step 5**

验证关系

```
MR(y, y')
```

如果违反：确认出现 bug
