### 3.prompt cache

  Prompt cache 命中的前提是"每一次请求发给模型的前缀内容（system prompt + 历史消息）尽量保持逐字不变"。
  谁动的越少，命中率越高。

  OpenClaw 的做法：尽量不动

  它的历史管理策略叫 cache-ttl——只有缓存过期（默认 1 小时）之后，才会去修剪/压缩老的消息。也就是说：
  - 同一段对话只要还在 1 小时内，历史就一字不改
  - 即使要剪，也只剪「真的超大」（>5 万字符）的 tool 输出
  - 身份、SOUL、skills 这些都是冷加载一次就固定

  结果就是：下一轮请求的前缀跟上一轮几乎完全一样 → cache 高命中。

  Nanobot 的做法：每轮都在改

  它的压缩策略是按预算触发——只要上下文占到 60%/70%/85%，每一轮都会回头去改老消息：
  - 老的 tool 输出被改写成「已转存」「已精简」「重复输出」
  - 老的 assistant 调用参数被截断
  - 到 85% 还会直接删一批老消息，并在对话开头插入一段新的"上下文索引"
  - 另外 system prompt 是每次重新拼的，memory、skills、bootstrap 文件只要中途有变动，整个 system 段都会跟着变

  每一处改动 → 缓存前缀字节就变 → 下一次请求就 miss，等于反复重写缓存。

  一句话总结

  OpenClaw 以"保护缓存"为核心，只在必要时动历史；Nanobot 以"省 token / 不爆上下文"为核心，每轮都主动改历史。
  前者命中率天然就高，后者是用缓存命中率换上下文容量。

  要缩小差距，方向是把"主动压缩"改成"被动、延迟、不可逆"——比如只在真正快爆时动一次、不做逐轮去重、把易变的内容（memory、时间、运行信息）从 system 挪到最新的 user 消息里，让前面的稳定部分真正稳下来。
