## 三、建议的开发流程：按 4 个阶段推进

不要一上来就把所有技术都接满。  
正确顺序是：**先跑通最小闭环，再做增强。**

---

### 第一阶段：最小可用版本（MVP）

目标：先让聊天系统“能发、能收、能存”。

功能范围：

- 用户登录，返回 JWT
    
- 建立 WebSocket 连接
    
- 单频道聊天
    
- 消息写入 MySQL
    
- 查询频道历史消息
    
- 在线用户列表
    

这一阶段先**别急着上 RabbitMQ**，甚至 Redis 也可以先只做最简单在线状态。  
先把主干流程跑通：

`登录 -> 建立连接 -> 发送消息 -> 广播给频道成员 -> 消息存库 -> 拉历史消息`

你们第一周应该只做这个。

**阶段产出：**

- 可以演示的后端接口
    
- Postman / Apifox 文档
    
- 数据表初稿
    
- WebSocket 消息协议初稿
    
- README 初版
    

---

### 第二阶段：状态与缓存增强

目标：解决“谁在线、谁在频道里、未读数怎么记”。

新增：

- Redis 存在线状态
    
- Redis 存频道成员列表
    
- Redis 存未读消息数
    
- 用户上线/下线状态同步
    
- 加入频道/退出频道
    
- 拉取未读数
    

这一阶段要想清楚：

- 在线状态以什么为准，WebSocket session 还是心跳
    
- Redis 和 MySQL 哪些是强一致，哪些允许最终一致
    
- 未读数是实时更新还是按事件累计
    

---

### 第三阶段：异步化与解耦

目标：把高频操作从主链路中拆出去。

新增：

- RabbitMQ 异步落库
    
- 频道事件广播
    
- 可能的系统通知
    
- 重试 / 死信 / 消费异常处理
    

这时再引入 MQ 最合适，因为你们已经知道系统哪里慢、哪里耦合。  
否则一开始上 MQ，很容易“技术用了，但价值说不明白”。

主链路可以这样讲：

- WebSocket 收到消息
    
- 服务先做基础校验
    
- 写入 Redis / 内存态做快速响应
    
- 投递 MQ
    
- 消费者异步落库
    
- 广播频道事件
    

不过这里要注意：  
如果是课程/简历项目，**别为了炫技搞过度复杂**。你们可以明确写成：

- 聊天消息采用异步落库
    
- 在线状态同步走 Redis
    
- 广播事件通过 MQ 解耦
    

重点是解释清楚“为什么”，而不是堆名词。

---

### 第四阶段：工程化完善

目标：让项目更像一个可讲、可维护的作品。

补充：

- 统一异常处理
    
- 参数校验
    
- 日志规范
    
- 配置分环境
    
- Docker Compose
    
- 接口文档
    
- 单元测试 / 集成测试
    
- 压测脚本
    
- README、架构图、时序图
    

这部分对简历和面试很加分。很多项目其实功能差不多，差距就在工程化表达。
