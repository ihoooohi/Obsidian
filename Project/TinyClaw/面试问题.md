## 为什么用 go？

事情是这样的，我在我的 2c2g的服务器上安装了 openclaw 但我发现占的内存太大了，500mb 吃不消，OpenClaw 的官方文档里常见的问题 FAQ 中说了 Node js gateway的大小在500mb 左右
，然后我研究了一下发现，TypeScript 本质运行在 Node.js 上，而 Node.js 依赖 V8 VM。 启动时会加载 VM、很多运行时模块，所以即使空载也会有比较高的内存。

刚好那段时间我在学习 go 的相关知识，知道 go 占内存小，Go 是编译成原生二进制运行，所以他没有 VM，也没有大量运行时模块，所以内存会很低。于是我打算用 go 来重构一个，最后他运行的 gateway 只要 32mb 大大减小。

不过后来我想了想 go 的优势不仅在于省内存，还有启动快，可移植（ARM，X86 架构）任意 Linux 设备都能sftp 到服务器直接跑，可移植性这一块，直接去我 github 上的 release 里下载二进制文件，然后 sftp 到服务器就能直接跑了，不需要安装 node.js

而且我觉得 go 编写的优势在低配硬件上很吃香，比如树莓派，老旧的安卓手机（ARM），我最近关注到一个 github 项目叫 picoclaw 就是，所以后续我会去 picoclaw 上提交 pr，而不是继续我的开发，或者将一些我的开发提交pr 到 picoclaw 上

## 为什么用bus

bus 的设计很符合 go 的并发哲学 ==*"不要通过共享内存来通信，要通过通信来共享内存"*==

- TinyClaw = 回调模式 ，收到消息立即调用传入的函数处理
	- "把消息处理的逻辑注入到 channel 中，当 channel 收到飞书消息时，自动调用你注册的处理函数"
- MyClaw = 中介者模式 ，消息先写入 Bus，由 Bus 分发给订阅者

1.同步改成异步，不用等待 ai 响应完，ack 不会超时不会发两遍
2.也就是新增一个 channel，不需要再写消息的处理函数
3.可以消息积压

用户实际体验: 一直收不到回复，或者收到重复回复

1.故事是这样的，一开始我是这么处理的消息接受和处理的，通过回调模式，收到消息立即调用传入的函数处理，但是后续使用我发现有问题，我发一次消息会返回我两次回答或者就不回超时断开连接了，后面我发现是因为这是同步的，没有返回 ack，飞书又自动发送了一次消息，也就是说返回 ack 这一步被阻塞了

2.后面我尝试将处理消息的函数handler 异步（函数前加一个 go），直接不阻塞可以返回了（return nil，函数的返回值是 error），这样就解决了返回两次消息的问题，但是又有新的问题，如果连续发送多条消息，因为处理消息的函数异步了，返回顺序就不确定了，就乱序了

3.最后的解决办法是引入 bus，飞书消息写入 bus 直接返回（不等处理），Inbound接收消息，串行一条条处理，outbound 接收返回值，DispatchOutbound方法监听 outbound，将消息返回给用户，两者的select case 监听 channel 后的处理都是同步的，必须是一条条处理，保证了返回顺序，Inbound channel 带缓冲不阻塞立马返回 200，消息不会重发，完美解决这两个问题

```go
Inbound <- msg //Inbound是带缓冲的 channel，不阻塞
WriteHeader(http.StatusOK) 

//Inbound
select{
	case msg := <- Inbound:
		result,err := g.runAgent(ctx, msg)//g 是 gateway
		Outbound <- result
	
}

//Outbound
select{
	case msg := <- Outbound:
	send(msg)
}
```
