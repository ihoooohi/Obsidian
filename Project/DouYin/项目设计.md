# 一、JWT 鉴权设计

## 1. 先理解 HTTP 的基本问题

**HTTP 是无状态的** —— 这句话的意思是：

```
服务器：我不管你是谁，每次请求我都当成新用户
用户：   ？？？那我每次登录完，下次访问你又不认识我了？
```

比如：
1. 你登录了淘宝
2. 你刷新页面
3. 服务器：你是谁？不认识，点外卖去！

---

## 2. 最早的解决方案：Session（会话）

```
用户登录 → 服务器生成一个"会话ID"存服务器内存 → 返回给浏览器

浏览器 → 以后的请求都带着这个 ID
服务器 → 看到 ID 就知道"哦，这是刚才那个用户"
```

**缺点**：服务器需要存几亿个用户的会话ID，内存压力大。而且如果服务器重启，所有人需要重新登录。

---

## 3. 现代方案：JWT（JSON Web Token）

JWT 的核心思想：**服务器不存东西，让客户端自己带"身份证"**

```
用户登录 → 服务器用密码+时间+密钥 生成一个"加密字符串"（JWT）
           这个字符串里包含用户信息，服务器不存

浏览器 → 以后每次请求都带上这个字符串

服务器 → 用密钥解密一看："哦，是 userID=123 的小明"
```

这个 JWT 就像一张**防伪证书**：
- 有签名，无法伪造
- 包含用户信息
- 附带过期时间

---

## 4. JWT 的致命问题：无法"作废"

这是最关键的一点！

```
场景：用户改密码了

方案A（无 JWT 撤销）：旧 token 依然能用，直到 24 小时后过期
                   → 用户改密码后，旧手机依然能登录 24 小时！不安全！

方案B（这个项目的做法）：服务器存一份 token 比对
                      → 改密码时清空 token → 旧 JWT 立即失效
```

---

## 5. 这个项目具体怎么做的？

```
┌─────────────────────────────────────────────────────────────┐
│  登录时：                                                    │
│  1. 验证密码正确                                            │
│  2. 生成 JWT ──────────────────────────────────────────┐   │
│  3. 把 JWT 存入 MySQL（account.token 字段）            │   │
│  4. 把 JWT 存入 Redis（高性能）                         │   │
│  5. 返回给浏览器                                          │   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  验证请求时：                                                │
│  1. 浏览器带上 JWT（Authorization: Bearer xxx）            │
│  2. 服务器解密 JWT，得到 userID=123                       │
│  3. 查 Redis/MySQL："userID=123 的 token 是多少？"         │
│  4. 比对：如果一样 → 放行    不一样 → 拒绝                │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│  登出/改密码时：                                             │
│  1. 把数据库里的 token 清空（设为空字符串）                │
│  2. 下次验证时，比对失败 → 拒绝登录                        │
│  → 旧 JWT 立即失效！                                       │
└─────────────────────────────────────────────────────────────┘
```

---

## 6. 为什么存两份？Redis + MySQL

| 存储 | 用途 | 优点 | 缺点 |
|------|------|------|------|
| **Redis** | 主存储 | 极快，内存数据库 | 可能重启丢失 |
| **MySQL** | 兜底 | 持久化，不会丢 | 慢一些 |

验证顺序：**先查 Redis → 快了；如果 Redis 挂了 → 查 MySQL 兜底**

---

## 7.该项目的两种鉴权

| 模式 | 用途 | 区别 |
|------|------|------|
| **JWTAuth** | 强制登录（点赞、评论等） | 无Token直接返回401 |
| **SoftJWTAuth** | 可选登录（刷Feed流） | 无Token也放行，继续处理 |

| 模式          | 无 Token 时的行为              | 代码                         |
| ----------- | ------------------------- | -------------------------- |
| JWTAuth     | 直接返回 401，**不调用** c.Next() | c.AbortWithStatusJSON(...) |
| SoftJWTAuth | **调用** c.Next() 放行        | c.Next()                   |

## 8.JWT组成

```
JWT 就像一张"电子身份证"或"演唱会门票"

组成:  header.payload.signature (三段式，用 . 分隔)
例子:  eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwidXNlcm5hbWUiOiJhZG1pbiIsImV4cCI6MTYyMDAwMDAwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
        ───────────────────────────────────────────────────────────────────────────────
        header: 声明类型和算法
                          ─────────────────────────────────────────────────
                          payload: 存放用户信息 (可读)
                                                        ──────────────────
                                                        signature: 签名/防伪
```

**JWT 存放的信息（Payload）**：
```json
{
  "accountID": 12345,
  "username": "zhangsan", 
  "exp": 1699999999  // 过期时间
}
```

**为什么用 JWT？**
- **无状态**: 服务端不需要存储 token，节省数据库开销
- **跨域**: 适合前后端分离、移动端等场景

# 三、RabbitMQ设计

## 1.消息队列（MQ）基础

### 什么是消息队列？

**消息队列**（Message Queue）是一种**异步通信协议**/技术，用于在不同的应用程序、服务或系统组件之间传递消息。

### 核心概念

| 概念                | 说明           |
| ----------------- | ------------ |
| **Producer（生产者）** | 发送消息的一方      |
| **Consumer（消费者）** | 接收并处理消息的一方   |
| **Queue（队列）**     | 消息存储的容器，先进先出 |
| **Broker（中间件）**   | 消息队列的管理服务    |
|                   |              |

---

## 2.RabbitMQ 是什么？

### 概述

**RabbitMQ** 是一个实现了 **AMQP（高级消息队列协议）** 的开源消息中间件

### 核心概念

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Producer   │────▶│   Exchange  │────▶│    Queue    │────▶│  Consumer   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  Binding    │
                    │  (routing)  │
                    └─────────────┘
```

| 组件 | 说明 |
|------|------|
| **Exchange（交换机）** | 接收生产者消息，根据规则转发到队列 |
| **Queue（队列）** | 存储消息 |
| **Binding（绑定）** | 交换机和队列之间的关联规则 |
| **Routing Key（路由键）** | 消息的"地址"，用于交换机判断发给哪个队列 |

### Exchange 类型

| 类型 | 说明 |
|------|------|
| **Direct** | 完全匹配路由键 |
| **Topic** | 支持通配符匹配（`*`匹配一个词，`#`匹配零个或多个词） |
| **Fanout** | 广播给所有绑定的队列 |
| **Headers** | 根据消息头属性匹配 |


# worker并发错误处理设计

## 代码解析

### 1. 创建错误通道

```go
errCh := make(chan error, 4)
```

- 创建了一个 **带缓冲的 channel**，容量为 4
- 缓冲大小为 4 是因为最多会启动 4 个 worker goroutine
- 使用缓冲 channel 可以避免发送方阻塞，因为主 goroutine 只会接收一次错误

### 2. 启动 Worker Goroutines

```go
go func() { errCh <- socialWorker.Run(ctx) }()
go func() { errCh <- likeWorker.Run(ctx) }()
go func() { errCh <- commentWorker.Run(ctx) }()
if popularityWorker != nil {
    go func() { errCh <- popularityWorker.Run(ctx) }()
}
```

这里启动了 **4 个并发的 goroutine**，每个 goroutine 运行一个 worker：

| Goroutine | Worker | 作用 |
|-----------|--------|------|
| 1 | SocialWorker | 处理关注/取关等社交事件 |
| 2 | LikeWorker | 处理点赞/取消点赞事件 |
| 3 | CommentWorker | 处理评论事件 |
| 4 | PopularityWorker | 更新视频热度（可选，如果 Redis 可用） |

每个 worker 的 `Run(ctx)` 方法会阻塞运行，直到收到消息或上下文取消。

### 3. 接收错误

```go
err = <-errCh
```

- 主 goroutine **阻塞等待** 任意一个 worker 返回错误
- 一旦某个 worker 发生错误（或者上下文被取消），就会向 `errCh` 发送错误
- 主 goroutine 收到错误后，整个程序会退出

### 整体流程图

```
┌─────────────────────────────────────────────────────┐
│                      main goroutine                  │
│                                                     │
│   errCh := make(chan error, 4)  ←── 容量为4的缓冲   │
│                    ↓                                │
│              阻塞等待: err = <-errCh                │
│                    ↓                                │
│              收到错误 → 打印日志 → 退出              │
└─────────────────────────────────────────────────────┘
          ↑          ↑          ↑          ↑
          │          │          │          │
    ┌─────┴──┐  ┌────┴──┐  ┌───┴──┐  ┌────┴────┐
    │Social  │  │Like   │  │Comm- │  │Popularity│
    │Worker  │  │Worker │  │ent   │  │ Worker   │
    │ goroutine │ goroutine │ goroutine │ goroutine │
    └─────────┘  └────────┘  └────────┘  └──────────┘
          ↑          ↑          ↑          ↑
          └──────────┴──────────┴──────────┘
                     RabbitMQ 消息队列
```

### 设计要点

1. **错误传播**：使用 channel 将子 goroutine 的错误传递回主 goroutine，统一处理
2. **优雅关闭**：通过 `signal.NotifyContext` 监听系统信号，任何中断都会传播到所有 worker
3. **并发安全**：多个 worker 并发运行但互不干扰，通过 channel 进行协调
