# 百万并发怎么办

## 百万级并发架构思路

### 一、流量分层拦截

```
用户请求
    ↓
[CDN] - 静态资源、热点数据缓存
    ↓
[负载均衡 Nginx] - 限流、熔断
    ↓
[网关层] - 鉴权、限流、路由
    ↓
[应用集群] - 水平扩展
    ↓
[缓存层 Redis] - 热点数据
    ↓
[数据库] - 主从分离
```

**思路**：把流量挡在前面，层层过滤，不要让请求直接打到数据库

---

### 二、具体手段

| 层级 | 手段 | 作用 |
|------|------|------|
| **CDN** | 静态资源、首页热点数据 | 挡掉 50%+ 请求 |
| **Nginx** | 限流、连接复用、keepalive | 基础防护 |
| **网关** | 统一鉴权、限流、熔断 | 入口保护 |
| **应用** | 无状态部署、连接池 | 水平扩展 |
| **缓存** | 多级缓存、本地+Redis | 减少 DB 压力 |
| **DB** | 主从、读写分离、分库分表 | 数据层扩展 |

---

### 三、百万并发的关键点

**1. 无状态设计**
- 应用层无状态，可以无限水平扩展
- 状态放到 Redis（Session、Token）

**2. 缓存为王**
- 90% 请求走缓存
- 热点数据预加载
- 缓存雪崩/击穿/穿透解决方案

**3. 异步化**
- 写操作 MQ 削峰
- 消息队列积压能力：百万级/秒
- 你项目里的 RabbitMQ 就是这个作用 ✅

**4. 限流熔断**
- 令牌桶 / 滑动窗口
- 熔断降级：服务挂了有兜底
- 你的限流器是入门级 ✅

**5. 数据库扩展**
- 读写分离：主从同步
- 分库分表：按用户ID hash
- 连接池复用

---

### 四、你项目可以升级的方向

| 当前 | 升级到百万级 |
|------|-------------|
| 单机 Redis | Redis Cluster / 主从 |
| 单机 MySQL | 主从 + 分库分表 |
| 本地限流 | Nginx 层限流 + 网关限流 |
| 同步调用 | 异步消息队列 |
| 无监控 | Prometheus + Grafana |

---

### 五、面试话术

> "百万级并发是一个系统工程，不是某一个点优化就能解决的。我会从以下几个层面考虑：
> 
> 1. **流量分层**：CDN → Nginx → 网关 → 应用 → 缓存 → DB，层层过滤
> 2. **水平扩展**：应用层无状态，可以加机器
> 3. **异步化**：写操作走 MQ 削峰，跟我项目中用 RabbitMQ 一样
> 4. **限流熔断**：防止系统被突发流量打垮
> 5. **数据分层**：缓存挡掉 90% 请求，数据库只处理 10%
> 
> 具体到我的项目，目前用的是单机 Redis + MySQL，如果要扩展到百万级，需要引入 Redis Cluster、主从分离，以及在 Nginx 层做限流。"


# 请说说分布式锁的实现？

项目在视频详情页使用了SETNX分布式锁来解决缓存击穿问题。

实现原理 ：使用Redis的SETNX命令原子加锁，同时设置2秒TTL防止死锁。每个锁对应一个唯一token（16位随机字符串），解锁时用Lua脚本校验token，只有持有正确token才能删除锁，防止误删。

使用流程 ：当缓存未命中时，多个并发请求只有一个能获取锁，其他请求进入轮询等待（最多100ms）。获取锁的请求会双重检查缓存（可能已被其他协程回填），确认无缓存后回源MySQL构建缓存。

效果 ：将1000个并发请求减少为1个回源请求，其他999个从缓存读取，完美解决了热点key失效时的击穿问题。

# 滑动窗口热榜排名怎么做的

```text
查询时刻: 14:30

要合并的60个桶:
┌─────────────────────────────────────────────────────────────┐
│ 14:30桶 │ 14:29桶 │ 14:28桶 │ ... │ 13:31桶                 │
│ {C:+5}  │ {B:+8}  │ {A:+2,  │     │ (更早的)                 │
│         │         │  B:+2}  │     │                         │
└─────────────────────────────────────────────────────────────┘
                            ↓ ZUNIONSTORE (SUM)
              ┌─────────────────────────────────┐
              │ 合并结果 (临时key):               │
              │ 视频A: 2分                       │
              │ 视频B: 8+2=10分                   │
              │ 视频C: 5分                       │
              └─────────────────────────────────┘
                            ↓ ZREVRANGE (倒序取)
              排名: B(10分) > C(5分) > A(2分)
```
