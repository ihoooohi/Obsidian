### 📅 Day 2：JWT鉴权 + 缓存自愈机制（面试必问！）

**目标**：能详细解释Token验证流程

**面试话术**：
```
"我的项目使用JWT Token鉴权，实现了'缓存自愈机制'：

1. 登录成功后，Token存两份：
   - MySQL（持久化）
   - Redis account:{id}（高性能）

2. 验证Token时：
   - 先查Redis，命中直接用
   - Redis没有/失败 → 回退查MySQL
   - 查到后回填Redis（自愈）

3. 两种鉴权模式：
   - JWTAuth：强制登录（如点赞）
   - SoftJWTAuth：可选登录（如刷Feed）"
```

### 📅 Day 3：点赞流程 + MQ异步处理（最加分！）

**目标**：能完整画出点赞流程图

**核心流程**：
```
用户点赞 
  → 检查视频存在 + 检查是否已点赞
  → 尝试发MQ消息（2个队列）
     ├─ like.events → Worker消费 → 写MySQL
     └─ popularity.events → Worker消费 → 更新Redis热榜
  → 如果MQ失败 → 降级直写（直接写MySQL/Redis）
  → 返回成功
```

**面试话术**：
```
"点赞流程我实现了'异步解耦 + 降级保证'：
- 用户点赞后立即返回，不用等数据库写入
- 通过RabbitMQ实现异步写，削峰填谷
- 如果MQ不可用，立即降级为直写，保证数据不丢

这样设计：用户体验好（响应快）+ 高可用（MQ挂了也能工作）"
```

### 📅 Day 4：Feed流 + 分页策略（体现技术深度！）

**目标**：能说出4种Feed流的区别

**4种Feed流**：

| 接口               | 排序方式  | 分页方式               | 需要登录 |
| ---------------- | ----- | ------------------ | ---- |
| listLatest       | 时间倒序  | 时间游标               | 可选   |

**面试话术（复合游标）**：
```
"点赞数排序时，可能100个视频都是10个赞，
单纯按点赞数分页会重复。

我的方案：'点赞数+ID'复合游标
WHERE likes_count < ? OR (likes_count = ? AND id < ?)

保证：下一页的第一条 < 上一页的最后一条
实现：不重复、不遗漏"
```

### 📅 Day 5：Redis缓存策略（亮点的核心！）

**4大缓存策略**（必须能详细说出来）：

| 策略 | 场景 | 实现 |
|------|------|------|
| 缓存自愈 | Token验证 | Redis失败→MySQL→回填Redis |
| 防击穿 | Feed缓存 | SETNX互斥锁，只一个请求回源 |
| 滑动窗口热榜 | 热榜 | 按分钟分桶ZSET + ZUNIONSTORE聚合 |
| 主动失效 | 数据变更 | 删视频→DEL相关缓存 |
