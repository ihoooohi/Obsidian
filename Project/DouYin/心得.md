## 自动“热启动”缓存（自愈机制）

  这是最巧妙的一点。当 Redis 宕机恢复（或者刚重启，里面是空的）时：
   * 冷启动问题：通常缓存重启后，大量请求瞬时涌入 DB（缓存穿透/击穿），容易把 DB 打挂。
   * 自愈逻辑：
       1. 第一个请求进来，Redis 没中，查 DB 验证通过。
       2. 关键动作：验证通过后，代码会顺手把这个 Token 重新写回 Redis（cache.SetBytes）。
       3. 第二个及以后的相同用户请求进来，就能直接命中 Redis 了。
   * 结果：缓存不需要手动预热，而是随着用户的自然请求，自动、分散地完成填充。

## 为什么用JWT 存redis存数据库，不用 session（会话 id） 存 redis 存数据库

首先想到只用 jwt，但想到 jwt 不安全，无法撤销，用户改密码旧的 token 还能用，就把 jwt 存数据库里，那既然都存数据库了为什么不用 session token，因为攻击者可以伪造多个 session id（比如 uuid）这样后端必须每次都去查 Redis（即使是无效ID）。如果 Redis 没命中，还得去查 DB。这会导致数据库被无效查询压垮。 发生==**缓存穿透**==

但是 jwt 不会，cpu 层面就拦截了，后端有验证功能，不会进到数据库

jwt 存数据库相当于双重保险

这种 “JWT 表现层 + Session 逻辑层” 的结合，是规避纯 JWT 无法撤销、纯 Session 压力过大这两个缺点的工程化套路。

**改进方案：** 用 jwt access token + refresh token 存 redis  但这个方案只是解决了可撤销，没解决 refresh token id 大量攻击的问题，所以还要加上 **refresh  ip限流 **
