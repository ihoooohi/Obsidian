
# feed
## v1

\textbf{feedsystem\_video\_go -- Go语言高性能短视频 Feed 流平台} \hfill 2025年11月 -- 2026年2月 \\
\textbf{项目描述：} 基于开源项目进行二次开发与架构优化，实现短视频社交 Feed 流系统，提供账号、视频、点赞、评论、关注等核心功能，采用 Gin 构建高性能 Web 服务并支持 Docker 部署。 \\
\textbf{技术栈：} Go \;|\; Gin \;|\; GORM \;|\; MySQL \;|\; Redis \;|\; RabbitMQ \;|\; Docker
\begin{itemize}
    \item 设计\textbf{高并发 Feed 流架构}，支持匿名流与登录态个性化推荐双模式，采用\textbf{游标分页}替代 Offset 分页，保证大数据量场景下分页稳定性与查询性能。
    \item 实现 \textbf{JWT + Redis 双层鉴权缓存机制}，支持\textbf{强制下线策略}（密码修改后旧 Token 立即失效），通过缓存自愈机制减少频繁回源 MySQL。
    \item 构建\textbf{多级缓存体系}（Token / Feed流 / 视频详情 / 热榜 ZSET），使用 \textbf{SETNX 分布式锁}解决缓存击穿问题。
    \item 设计\textbf{热榜滑动窗口算法}（分钟分桶 + 聚合计算），降低实时统计计算成本并提升排行榜更新效率。
    \item 基于 \textbf{RabbitMQ 实现事件驱动架构}，将点赞、评论、社交事件\textbf{异步化处理}，解耦写扩散与热度计算；MQ 异常时自动降级为直写数据库以保证数据一致性。
\end{itemize}

## v2
\textbf{feedsystem\_video\_go -- Go语言高性能短视频 Feed 流平台} \hfill 2025年11月 -- 2026年2月 \\
\textbf{项目描述：} 在开源仿TikTok项目基础上进行架构重设计与功能扩展，独立完成缓存体系、
异步架构、鉴权模块的设计与实现，提供账号、视频、点赞、评论、关注等核心功能，
支持 Docker Compose 一键部署。 \\
\textbf{技术栈：} Go \;|\; Gin \;|\; GORM \;|\; MySQL \;|\; Redis \;|\; RabbitMQ \;|\; Docker
\begin{itemize}
    \item \textbf{【性能】} Feed流在高频读场景下存在大量 MySQL 全表扫描与 Offset 分页性能退化问题；
    引入\textbf{游标分页}替代 Offset、构建\textbf{四层 Redis 缓存}（Token / Feed流 / 视频详情 / 热榜 ZSET），
    并以 \textbf{SETNX 分布式锁}防止缓存击穿；压测显示 Feed 接口 QPS 从 \textasciitilde{}300 提升至 \textasciitilde{}3200，
    P99 延迟降低约 \textbf{82\%}。

    \item \textbf{【架构】} 点赞/评论/热榜计算与主请求链路强耦合，导致写接口 TP99 随负载线性劣化；
    基于 \textbf{RabbitMQ} 将社交写事件完全异步化，拆分出独立 Worker 进程消费热度计算任务，
    设计\textbf{滑动窗口分钟分桶}聚合算法降低统计开销；MQ 不可用时\textbf{自动降级直写 DB} 保证数据不丢失，
    写接口平均耗时降低约 \textbf{60\%}。

    \item \textbf{【安全】} 传统单 Token 方案无法支持密码修改后旧会话立即失效；
    设计 \textbf{JWT + Redis 双层鉴权}，Token 有效性由 Redis 版本号二次校验，
    密码变更后旧 Token 版本号失效实现\textbf{强制下线}；缓存自愈机制在 Redis 冷启动后
    自动回源重建，减少服务重启期间 MySQL 突发压力约 \textbf{70\%}。
\end{itemize}

## v3
\textbf{feedsystem\_video\_go -- 短视频 Feed 流系统架构优化} \hfill 2025年11月 -- 2026年2月 \\
\textbf{项目描述：} 围绕短视频 Feed 场景，对开源项目进行二次开发与关键链路优化，重点解决分页稳定性、鉴权一致性和互动写入削峰问题。 \\
\textbf{技术栈：} Go \;|\; Gin \;|\; GORM \;|\; MySQL \;|\; Redis \;|\; RabbitMQ \;|\; Docker
\begin{itemize}
    \item 面向“匿名浏览 + 登录态互动状态回填”场景，设计 \textbf{SoftJWTAuth + JWTAuth} 双鉴权模式，并将最新流、点赞榜、热榜分别抽象为 \textbf{latest\_time}、\textbf{likes\_count + id}、\textbf{as\_of + offset} 三类分页策略，解决热度实时变化下的重复、漏数与跳页问题。
    \item 针对热点 Feed 与鉴权链路的高频访问，构建 \textbf{Redis 多级缓存 + DB 兜底自愈} 方案；Feed/关注流与视频详情缓存采用互斥锁防击穿，降低并发回源对 MySQL 的冲击。
    \item 将点赞、评论、关注与热视频热度更新改造为 \textbf{RabbitMQ + Worker} 事件驱动链路，接口层仅负责校验与事件发布；MQ 发布失败时同步降级直写 MySQL/Redis，兼顾响应时延与数据一致性。
    \item 在 \textbf{[并发数]} 压测下，\texttt{/feed/listLatest} 的 p95 延迟从 \textbf{[A]ms} 降至 \textbf{[B]ms}，MySQL 查询量下降 \textbf{[C]\%}，缓存命中率达到 \textbf{[D]\%}。
    \item 通过 Redis / RabbitMQ 故障注入验证核心链路可用性：登录、Feed 浏览、点赞等接口成功率保持在 \textbf{[E]\%+}；同时借助 Docker Compose 将本地环境搭建时间从 \textbf{[F]分钟} 缩短至 \textbf{[G]分钟}。
\end{itemize}

## v4
\begin{itemize}
    \item \textbf{读路径优化：} Feed 流接口存在两个性能瓶颈——Offset 分页在深翻页时需全表扫描
    导致查询随页数线性劣化，以及缓存缺失时高并发读请求穿透直打 MySQL。
    将分页策略改为\textbf{游标分页}（以上次返回的最大 ID 为基准定位，查询复杂度从 O(N) 降至 O(1)）；
    同时构建\textbf{四层 Redis 缓存}（Token / Feed 列表 / 视频详情 / 热榜 ZSET），
    以 \textbf{SETNX 分布式锁}防止缓存击穿时的惊群效应。
    压测结果（1200 VU，10 分钟）：Feed 接口 QPS \textbf{从 $\sim$300 提升至 $\sim$13700}，P95 延迟降低约 \textbf{88\%}。

    \item \textbf{写路径解耦：} 点赞、评论、热榜计算若同步落库，写高峰期 DB 压力会反向拖慢读接口响应。
    引入 \textbf{RabbitMQ} 将所有社交写事件异步化，拆分独立 Worker 进程消费热度任务，
    设计\textbf{分钟级滑动窗口分桶}聚合算法替代实时计算；MQ 不可用时\textbf{自动降级直写 DB}，
    保证数据不丢失。读写混合压测下（1500 VU），写接口 P95 延迟在读写混合压测下降低约 \textbf{80\%}，
    读接口响应时间在写并发峰值期保持稳定（抖动 $<$ 5\%）。
\end{itemize}

# TinyClaw

## v1
\textbf{TinyClaw -- 轻量级 AI Agent 服务（OpenClaw 架构方案）} \hfill 2026年2月 -- 至今 \\
\textbf{项目描述：} 基于 Go 语言实现的轻量级 AI Agent 服务，支持飞书即时通讯接入，用户可通过自然语言远程控制服务器执行操作；在 OpenClaw 架构基础上进行重构优化，移除冗余模块，使系统可部署于 2C2G 低配置服务器环境。 \\
\textbf{技术栈：} Go \;|\; Eino \;|\; WebSocket \;|\; LLM Agent \;|\; Docker
\begin{itemize}
    \item 设计并实现 \textbf{ReAct Agent 架构}，基于 Eino 框架构建 \textbf{Reasoning + Acting 推理循环}，支持多轮对话、工具调用与结果观察，实现自然语言驱动的服务器操作。
    \item 设计 \textbf{Channel / Gateway 分层架构}，实现 \textbf{飞书 WebSocket 长连接接入}，支持消息异步处理与快速 ACK 响应，提升系统并发处理能力。
    \item 构建\textbf{命令执行工具链}，实现 Shell 执行工具并加入\textbf{安全防护机制}（危险命令检测、执行超时控制、输出截断），保障远程运维操作安全性。
    \item 重构 OpenClaw 原始架构，移除冗余组件并优化资源管理，使系统\textbf{内存占用降低约 95\%}，可稳定运行于低配置云服务器。
\end{itemize}
