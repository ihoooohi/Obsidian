## 面试故事：抖音视频 Feed 系统性能优化

### 🎯 Situation（背景）

"我之前负责一个抖音视频 Feed 系统的后端开发。这是一个典型的社交类应用，核心功能包括：
- **Feed 流**：用户浏览视频列表
- **点赞/评论**：用户互动操作
- **热度计算**：视频热度实时更新

系统上线初期，我们用的是 **Baseline 架构**：
- 纯 MySQL 存储，无缓存
- 点赞/评论同步写数据库
- Feed 流直接查 DB

**问题**：压测显示，在 1200 并发用户下，Feed 接口 P95 延迟高达 **1.1 秒**，QPS 只有 **3200**，完全无法满足业务需求。"

---

### 📝 Task（任务）

"业务方要求：
1. Feed 接口 P95 延迟降低到 **200ms 以内**
2. 整体 QPS 提升 **3 倍以上**
3. 支持写操作（点赞/评论）的异步处理，不能阻塞用户请求"

---

### ⚡ Action（行动）

我主导了两次重大优化：

#### 第一阶段：Optimized 版本（引入缓存 + 消息队列）

**1. 引入 Redis 缓存**
- Feed 流优先读 Redis，缓存热门视频
- 点赞数、评论数缓存到 Redis，减少 DB 查询

**2. 引入 RabbitMQ 异步写**
- 点赞/评论操作改为：先写 Redis，再发消息到 MQ
- 后端 Worker 异步消费，批量写入 MySQL

**3. Worker 架构**
- 单进程、单 connection、单 channel、单 goroutine
- 四类队列：social、like、comment、popularity

**代码改造示例**：
```go
// 改造前：同步写 DB
func (s *Service) Like(ctx context.Context, req *LikeRequest) error {
    return s.db.Create(&Like{VideoID: req.VideoID}).Error  // 阻塞等待
}

// 改造后：异步写 MQ
func (s *Service) Like(ctx context.Context, req *LikeRequest) error {
    // 1. 先写 Redis（立即返回）
    s.redis.Incr(ctx, fmt.Sprintf("like:%d", req.VideoID))
    
    // 2. 发消息到 RabbitMQ（异步）
    return s.mq.Publish("like.events", "like.create", req)
}
```

#### 第二阶段：Refactor 版本（Worker 并发优化）

**发现问题**：Optimized 版本虽然读性能提升，但写延迟极高（Like P95 达到 **89 秒**），原因是 Worker 单 goroutine 串行消费，队列积压严重。

**解决方案**：
- **拓扑声明与消费分离**：单独 `topologyCh` 声明 Exchange/Queue
- **每 Consumer 独立 Channel**：避免共享 channel 阻塞
- **多 goroutine 并发消费**：
  - Like Worker：1 → 8 个 goroutine
  - Comment Worker：1 → 4 个 goroutine
  - Popularity Worker：1 → 8 个 goroutine
- **配置化 prefetch**：根据并发数动态调整 QoS

**核心代码**：
```go
func launchConsumerGroup(ctx context.Context, conn *amqp.Connection, 
    prefetch int, name string, queue string, concurrency int, ...) {
    
    for i := 0; i < concurrency; i++ {
        // 每个 consumer 独立 channel
        ch, err := openConsumerChannel(conn, prefetch)
        
        // 创建 worker
        r := worker.NewLikeWorker(ch, repo, queue)
        
        // 独立 goroutine 消费
        go r.Run(ctx)
    }
}
```

### ✅ Result（结果）

**性能对比**：

| 场景 | 版本 | 总 QPS | Feed P95 | Like P95 |
|------|------|--------|----------|----------|
| S1 纯读 | Baseline | 3,209 | 1,127ms | - |
| S1 纯读 | **Optimized** | **13,696** | **132ms** | - |
| S1 纯读 | **Refactor** | **15,137** | **113ms** | - |
| S2 混合 | Baseline | 2,954 | 2,092ms | 2,088ms |
| S2 混合 | Optimized | 6,710 | 569ms | 89,986ms ⚠️ |
| S2 混合 | **Refactor** | **7,715** | **439ms** | **3,595ms** ✅ |

**关键成果**：
1. **读性能**：Feed QPS 从 3,209 → 15,137，提升 **372%**；P95 从 1,127ms → 113ms，降低 **90%**
2. **写性能**：Like P95 从 89s → 3.6s，降低 **96%**
3. **架构升级**：从同步阻塞 → 异步消息队列 + 多并发消费

**经验总结**：
- 缓存是读性能提升的关键（Redis 扛住了 90% 读请求）
- 异步化是写性能的基础，但**并发消费**才是解决队列积压的核心
- 监控要到位，压测要覆盖各种场景（纯读、混合、写峰值）

### 💡 面试加分点

如果被追问，可以补充：

**Q：为什么 Worker 要每个 consumer 一个 channel？**
> "RabbitMQ 的 channel 不是线程安全的，多个 goroutine 共享一个 channel 会导致竞争和阻塞。每个 consumer 独立 channel 可以隔离故障，一个 channel 出问题不影响其他 consumer。"

**Q：prefetch 怎么设置？**
> "prefetch 要和并发数匹配。我们设置 prefetch=32，并发=8，这样每个 consumer 最多预取 32 条，既保证吞吐量又避免内存压力过大。"

**Q：如果还要优化，你会怎么做？**
> "下一步可以考虑：1) Worker 水平扩容，多实例部署；2) 队列分片，按 user_id 哈希到不同队列；3) 引入死信队列处理异常消息。"
