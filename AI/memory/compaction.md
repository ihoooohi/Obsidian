# Agent 记忆压缩策略

记忆压缩的核心问题是：**如何在有限的上下文窗口内，保留对当前任务最有价值的信息**。

## 常用策略

### 1. 滑动窗口（Sliding Window）

最简单的方案，只保留最近 N 轮对话，丢弃更早的内容。

- 优点：实现极简，延迟低
- 缺点：长任务中早期关键信息会丢失
- 适用：对话型 agent，短任务

### 2. 摘要压缩（Summarization）

定期把旧对话/事件批量交给 LLM 生成摘要，用摘要替换原始内容。

```
[摘要：用户在前10轮确认了目标是重构支付模块，已完成接口分析]
+ 最近5轮原始对话
```

- 触发时机：上下文超过阈值时、每 N 轮、任务阶段切换时
- 缺点：摘要本身有信息损耗，细节被抹平

### 3. 分层记忆（Hierarchical Memory）

将记忆分为不同层级，各层粒度不同：

|层级|内容|保留时间|
|---|---|---|
|工作记忆|当前步骤的完整上下文|短暂|
|情节记忆|关键事件的摘要|中期|
|语义记忆|提炼出的事实、偏好、规则|长期|

MemGPT 是这个思路的典型实现。

### 4. 重要性过滤（Importance Scoring）

给每条记忆打分，低分的优先被压缩或丢弃。评分维度：

- **新近性**：越新越重要
- **相关性**：与当前任务的语义相似度
- **访问频率**：被引用越多越重要
- **显式标记**：agent 主动标记为"重要"

Generative Agents（Stanford 小镇论文）里用的就是这套三维评分。

### 5. 结构化提取（Structured Extraction）

不存原文，而是把对话中的关键信息提取成结构化对象存起来：

```json
{
  "user_goal": "重构支付模块",
  "constraints": ["不能停服", "deadline 5月底"],
  "decisions_made": ["使用策略模式", "保留旧接口兼容层"],
  "pending": ["性能测试方案未定"]
}
```

压缩率高，但依赖提取质量，容易漏信息。

### 6. RAG 外挂记忆（Retrieval-Augmented）

把历史信息向量化存到外部数据库，每轮按相关性检索，而不是全量放进上下文。

- 优点：理论上记忆无限，按需取用
- 缺点：检索可能漏掉隐式关联；增加延迟和复杂度
- 适用：长期运行的 agent、知识密集型任务

### 7. 状态机 + 显式状态（Explicit State）

agent 维护一个结构化的"任务状态对象"，每步更新，而不是依赖对话历史来还原状态。

本质是把记忆的工作从"压缩历史"转移为"维护状态"，更可控。

# Anthropic & OpenAI
## Anthropic 的观点：Context Engineering

Anthropic 在 2025 年 9 月发了一篇博客 [Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)，核心论点是：

**"压缩记忆"本质上是一个 context engineering 问题，目标是找到能让模型产生期望行为的最小高信号 token 集合。**

他们提出了几个具体策略：

**1. 关注 Context Rot（上下文腐烂）** 随着 context 窗口里 token 数增长，模型准确召回信息的能力会下降。这源于 Transformer 的架构约束——n 个 token 之间存在 n² 的 pairwise 关系，context 越长，注意力就越被稀释。因此他们把 context 视为有限资源来管理，而不是无限堆叠。

**2. Just-in-time 上下文（按需加载）** 相比于预处理所有相关数据塞进上下文，agent 只维护轻量的标识符（文件路径、查询语句、链接等），在运行时通过工具动态加载数据。Claude Code 就是用这个思路处理大型数据库——用 bash 命令按需读取，而不是把整个数据对象塞进 context。

**3. System prompt 写在"正确高度"** 避免两个极端：一是把复杂 if-else 逻辑硬编码进 prompt（脆弱、维护复杂）；二是写得过于模糊、假设模型能自己理解背景。正确做法是：足够具体能指导行为，又足够灵活让模型有启发式空间。

**4. 用 examples 代替规则堆叠** Anthropic 建议避免把大量边界案例规则塞进 prompt，而是精心筛选一组多样的、典型的示例来表达期望行为。对 LLM 来说，examples 相当于"一图胜千言"。

## OpenAI Codex 的做法：持久化记忆层

Codex 走了一条不同的路，更偏向工程化的跨会话记忆系统：

Codex 的 Memories 功能让 agent 把过去会话里有用的上下文带到未来的任务中，包括稳定的用户偏好、重复的工作流、技术栈、项目规范和已知坑点，这样用户不需要在每个新会话里重复提供同样的背景。

主记忆文件存在 `~/.codex/memories/` 下，包含摘要、持久条目、近期输入和历史会话的支持证据。Codex 会跳过活跃或过短的会话，从生成的记忆字段中过滤掉 secrets，并在后台异步更新记忆而不是每次会话结束时立即写入。

此外 Codex 的 context 设计有个有意思的点：Codex 的 context 不只是对话历史，而是从多个来源动态组装的——仓库本身、`AGENTS.md` 文件（存在 repo 里的持久指令）、Skills（可复用的指令包）、当次任务 prompt、以及历史交互记忆。核心洞察是：Codex 的大部分上下文来自 repo 本身，而不是对话历史。

## 总结对比

| 对比项 | Anthropic | OpenAI Codex |
|--------|----------|--------------|
| 核心思路 | 最小高信号 token，just-in-time 加载 | 持久化记忆文件 + repo 作为上下文主体 |
| 记忆存储 | 外部引用标识符 + 工具动态检索 | ~/.codex/memories/ 文件系统 |
| 跨会话 | 依赖外部存储/工具 | 内置 Memories 功能 |
| 压缩策略 | 精简 prompt + 去掉低信号 token | 摘要 + 过滤 secrets + 异步写入 |

两家思路其实是互补的：Anthropic 更关注单次会话内的 context 质量，Codex 更关注跨会话的记忆持久化。实际工程中两者都需要。
