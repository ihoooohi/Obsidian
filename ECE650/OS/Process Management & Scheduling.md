## PCB(Process Control Block)

**为什么需要 PCB？**

现代系统里有几十上百个进程在"同时"运行，但 CPU 核心数有限。OS 需要不停地把进程换进换出 CPU。每次换出时，必须把这个进程"当前做到哪了"完整记录下来，下次换回来才能接着跑，而不是从头开始。这份记录就是 PCB。

## Process Schedule

**一句话概括这一章的逻辑：**

CPU 资源有限 → 需要调度 → 调度要有标准 → 各种算法各有取舍 → 现代 OS 用综合方案。---

**第一层：为什么需要调度？**

系统里活跃的进程/线程数，几乎永远多于 HW 线程数（即 CPU 核心数）。每个进程运行一段时间后，要么主动阻塞（等 I/O），要么被 OS 强制换下来（时间片到期）。OS 要决定"接下来让谁上 CPU"——这就是调度要解决的问题。

调度的两大目标：一是制造"多个程序同时跑"的假象（细粒度轮换），二是效率——趁一个进程等 I/O 的时候让另一个进程用 CPU，不让 CPU 空着。

---

**第二层：调度队列**

OS 用链表（PCB 链表）来管理不同状态的进程，主要有三类队列：新建队列（刚创建还没准备好）、就绪队列（随时可以上 CPU）、事件队列（在等某个 I/O 或事件，比如等键盘输入、等磁盘读完）。调度器每次从就绪队列里挑一个进程放上 CPU。

## Process Schedule Algrithm

评价一个调度算法用这几个指标：CPU 利用率、吞吐量、周转时间、等待时间、响应时间。下面逐一讲五种算法：

![[Pasted image 20260401110147.png]]

---

**FCFS** 最简单：谁先来谁先跑，跑完才轮到下一个。问题是如果第一个进程是个超长任务，后面所有人都得干等着。

**SJF(Shortest Job First)** 最聪明（理论上）：每次都挑"剩余时间最短"的进程跑，平均等待时间可以证明是最优的。但有两个硬伤——OS 根本不知道进程下一个 CPU burst 有多长（只能靠指数平均估计），而且短任务源源不断时，长任务可能永远等不到 CPU，这叫**饥饿（starvation）**。

**优先级调度** 是 SJF 的推广——SJF 其实就是"把 burst 长度当优先级"的优先级调度。饥饿问题同样存在，解决办法是 **aging**：进程在就绪队列里等的时间越长，优先级就自动升高，保证最终能被调度到。

**Round Robin** 专门为交互式系统设计：每人轮流拿一个时间片（如 10ms），时间到了不管你跑没跑完都必须让出 CPU。时间片选择是门学问——太长退化成 FCFS，太短则上下文切换的开销占主导。经验法则是 80% 的 CPU burst 应该短于时间片长度。

**多级反馈队列（MLFQ）** 是现代操作系统（Windows、macOS）实际用的方案。核心思路是"动态观察进程行为，据此调整优先级"：CPU 密集型进程（一直跑满时间片）被降到低优先级队列，I/O 密集型进程（频繁主动让出 CPU）被提到高优先级队列。低优先级队列用更大的时间片（跑的少但每次跑得久），防止 CPU 密集型进程被完全饿死。
