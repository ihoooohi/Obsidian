## 一、保护（Protection）

> [!info] 定义
> 保护是 OS 资源管理的**机制**，确保资源只能在 OS 授权下使用。
> 区别于**策略（Policy）**——策略定义规则，机制负责执行。
> **工程智慧：永远分离机制与策略。**

**保护的三大目标：**
1. 防止资源被未授权访问或滥用
2. 保护用户免受未授权访问造成的伤害
3. 确保进程只按允许的策略使用系统资源

### 核心原则：最小权限原则

> [!important] Principle of Least Privilege
> 用户、进程等只拥有**完成任务所需的最低权限**，不多给一点。
> 即"需要知道才告诉你（Need-to-know basis）"。

**好处**：某个组件失效或被攻破时，只影响系统中极少数的部分。

### 基本模型：主体、动作、权限、客体

| 要素 | 英文 | 含义 | 举例 |
|------|------|------|------|
| **主体** | Subject | 发起请求的实体 | 用户、进程 |
| **动作** | Action | 要执行的操作 | 读、写、删除、执行 |
| **权限** | Right | 主体对客体执行动作的许可 | 读权限 |
| **客体** | Object | 被操作的资源 | 文件、内存、设备 |

**客体的分类：**
- 硬件对象：CPU、内存区域、磁盘、键盘、显示器
- 软件对象：文件、目录、程序

> [!tip] UNIX 哲学
> "Everything is a file" —— 所有资源都可以统一用文件的方式管理权限。

### 访问矩阵（Access Matrix）
```
          File 1   File 2   File 3   File n
User 1     read    write      -       read
User 2    write    write    write      -
User 3      -        -        -       read
User m     read    write    read      read
```

- **行 = 主体（Subject）**
- **列 = 客体（Object）**
- **格子 = 权限（Rights）**

**实现方式：**

| 方式 | 描述 | 缺点 |
|------|------|------|
| 全局表（Global Table） | `<主体, 客体, 权限>` 三元组列表 | 表巨大、冗余多 |
| **访问控制列表（ACL）** | 每个对象维护 `<主体, 权限>` 列表（按列存储） | **推荐方式** |

### 动态权限

**两种实现方式：**

1. **修改客体的访问权限**
   - 例：`chmod` 命令改变文件的执行权限

2. **主体切换（Subject Switching）**
   - 例：`sudo` 临时切换为 root 身份
   - 例：UNIX 的 **setuid bit**
     - 每个文件有 owner ID 和 setuid 位
     - setuid = on：执行时进程的用户 ID 变为文件所有者的 ID
     - setuid = off：用户 ID 不变
     - 进程退出后恢复原 ID

> [!warning] 安全隐患
> 如果普通用户创建了一个 owner 为 root、且 setuid 为 on 的文件，会发生什么？
> → 任何执行该文件的用户都将获得 root 权限！

## 二、保护 vs 安全

| 维度 | 保护（Protection） | 安全（Security） |
|------|------|------|
| 针对对象 | **内部**用户和进程 | **外部**威胁和攻击者 |
| 关注点 | 本机程序和数据的访问控制 | 防止外部威胁绕过保护机制 |
| 举例 | 文件读写权限 | 防止密码泄露、防黑客入侵 |

> [!warning] 关键关系
> 保护做得再好，一旦安全被突破（如密码被盗），保护就形同虚设。
> 保护只有在用户按预期行为时才有效。

**安全的四个域：**
1. **物理安全**：限制机房物理访问
2. **人员安全**：防范社会工程学攻击、钓鱼
3. **OS 安全**：防范意外或蓄意的安全违规
4. **网络安全**：防止数据在传输中被截获或篡改

---

## 三、安全基础概念

| 术语 | 定义 | 举例 |
|------|------|------|
| **威胁（Threat）** | 安全违规的**潜在可能** | 程序中存在的漏洞 |
| **漏洞利用（Exploit）** | 通过代码/数据/命令利用漏洞**实施**攻击 | 打开网页越狱 iPhone |
| **攻击（Attack）** | **尝试**突破安全（可能成功或失败） | 运行一个 exploit |
| **恶意软件（Malware）** | 专门设计用于造成损害的软件 | 病毒、勒索软件 |

---

## 四、安全违规类别

| 类别 | 描述 | 举例 |
|------|------|------|
| **机密性破坏** | 数据窃取 | 盗取信用卡、账号信息 |
| **完整性破坏** | 未授权修改数据或代码 | 篡改银行账户余额 |
| **可用性破坏** | 销毁数据 | 删除客户信息、攻击网站 |
| **服务盗用** | 未授权使用资源 | 免费蹭 Netflix |
| **拒绝服务** | 阻止合法用户使用系统 | Ping flood 攻击 |

## 五、针对运行程序的威胁

### 特洛伊木马（Trojan Horse）

> [!note] 定义
> 伪装成合法软件的恶意代码，诱骗用户执行。**不会自我复制**。

**攻击原理（以 UNIX PATH 为例）：**
1. PATH 变量指定查找可执行文件的目录顺序，如 `/bin:/usr/bin:/usr/local/bin`
2. 若 PATH 中包含 `.`（当前目录）
3. 攻击者在某目录放一个名为 `cd` 的木马程序
4. 用户进入该目录执行 `cd`，实际执行了木马

> [!tip] 这就是为什么
> Linux 默认 PATH 不包含 `.`，所以运行自己编译的程序需要写 `./myprogram`

**典型案例：**
- **伪造登录界面**：捕获用户名和密码，然后显示"登录失败"并返回真实登录界面
  - Windows 的 `Ctrl+Alt+Delete` 就是为了防御此类攻击（该组合键无法被程序拦截）
- **间谍软件（Spyware）**：捆绑在正常软件中，暗中收集用户信息并通过隐蔽信道（Covert Channel）发送

### 后门（Backdoor）

> [!note] 定义
> 开发者**故意留在**合法软件中的安全漏洞，通常用于调试，但极易被遗忘和利用。

**最狡猾的变体：将后门藏在编译器中**
1. 编译器检测当前正在编译的是否是特定目标程序
2. 如果是，自动插入后门代码
3. 结果：审查源代码看不出任何问题，重新编译也无济于事！

**相关案例：** FlawedGrace 木马、"便士削减"式金融欺诈（通过后门将微小舍入误差汇集到指定账户）

### 逻辑炸弹（Logic Bomb）

> [!note] 定义
> 只在**满足特定条件时**才触发恶意行为的恶意软件，平时运行完全正常。

**触发条件示例：**
- 特定的年/月/日/时间
- 系统中存在特定信息
- 检测到开发者已离职

> [!example] 真实案例
> 西门子承包商 David Tinley 在交付软件中埋入逻辑炸弹，软件定期"故障"，然后他收费修复，持续两年后被发现。

### 栈溢出（Stack Buffer Overflow）

> [!important] 这是迄今为止最常见的攻击手段

**攻击原理：**

函数调用时的栈帧结构（从高地址到低地址）：
```
[ 函数参数 (a, b, c)      ]  ← 高地址
[ 返回地址 Return PC       ]  ← 关键！
[ 保存的帧指针 Frame Ptr   ]
[ 局部变量 buff1[0..4]    ]
[ 局部变量 buff2[0..9]    ]  ← 写入从这里开始
                               ← 低地址（新 SP）
```

**漏洞代码示例：**
```c
void func(int a, int b, char *c) {
    char buff1[5];
    char buff2[10];
    strcpy(buff2, c);  // 危险！没有边界检查
}
```

**攻击步骤：**
1. 向 `buff2` 写入超过 10 字节的数据
2. 数据溢出覆盖 Return PC
3. 将 Return PC 改为指向攻击者注入的代码
4. 函数返回时跳转执行恶意代码（如 `/bin/sh`）

**防御方法：**

| 方法 | 描述 |
|------|------|
| 安全编码 | 使用 `strncpy` 代替 `strcpy`，始终做边界检查 |
| 不可执行栈（NX） | 标记栈内存为不可执行 |
| 地址空间随机化（ASLR） | 随机化系统调用地址，难以预测跳转目标 |
| 栈金丝雀（Stack Canary） | 在返回地址前放置特殊值，检测是否被篡改 |

> [!tip] 延伸阅读
> "Smashing the Stack for Fun and Profit": https://insecure.org/stf/smashstack.html

### 病毒（Virus）

> [!note] 定义
> 嵌入在合法程序中的恶意软件，**会自我复制并主动传播**（这是与木马的核心区别）。
> 通常通过社会工程学传播（垃圾邮件、恶意下载）。

**病毒投递机制：** 病毒释放器（Virus Dropper）通常是木马，执行后将病毒注入系统。

**常见类型：**

| 类型 | 原理 |
|------|------|
| **可执行文件型** | 附加到文件末尾，修改程序入口跳转到病毒，执行后返回正常程序 |
| **引导扇区型（Boot）** | 感染 MBR，每次开机执行；也感染 USB 等可启动媒体；不出现在文件系统中 |
| **宏病毒（Macro）** | 嵌入在 Office 文件的宏代码中，打开文件即触发，并感染其他文件 |
| **源代码型** | 修改程序源码，在编译时传播 |

**规避检测的变体：**

| 变体 | 原理 |
|------|------|
| **多态病毒（Polymorphic）** | 每次安装时改变自身代码（特征码），躲避模式匹配 |
| **隐身病毒（Stealth）** | 拦截 `read()` 等系统调用，在杀软检查文件时隐藏自身 |
| **隧道病毒（Tunneling）** | 藏在中断处理程序或设备驱动中 |
| **加密病毒（Encrypted）** | 自带解密代码，运行时解密执行 |

## 六、针对系统和网络资源的威胁

### 端口扫描（Port Scanning）

> [!note] 定义
> 对目标主机的一段端口范围尝试 TCP/IP 连接，探测开放了哪些服务。

- 本身不一定是恶意行为（也用于正常的服务清单维护）
- 通常是攻击的**第一步**：发现可利用的脆弱服务（如旧版 sendmail）
- 成功连接后通常尝试缓冲区溢出攻击

> [!warning] 法律注意
> 在英国等国家，单纯执行端口扫描即可能构成犯罪！

### 蠕虫（Worms）

> [!note] 定义
> 本质上是**利用网络资源传播的病毒**，但不附着在其他程序上，可独立存在。

**结构：**
- **钩爪程序（Grappling Hook）**：最先在目标机器上建立立足点，然后从控制端请求下载蠕虫主体
- **载荷（Payload）**：主体恶意代码，可删除文件、勒索、窃取数据、安装后门，将主机变为"僵尸（Zombie）"

**传播方式：** 电子邮件、共享文件夹、恶意 HTML 脚本

> [!example] 莫里斯蠕虫（1988）
> - 作者：Cornell 研究生 Robert Morris（后成为 MIT 教授）
> - 首个互联网蠕虫，无实际载荷，但重复感染导致主机性能崩溃
> - 估计感染了 1/10 的互联网连接主机
> - 利用了三个漏洞：`rsh` 远程执行、`finger` 缓冲区溢出、`sendmail` 调试选项

### 僵尸网络（Bots / Botnet）

> [!note] 定义
> **Bot**：在被感染的"僵尸机器"上自主运行的恶意软件，无需机主同意。
> **Botnet**：由 Botmaster 通过通信信道统一控制的大量 Bot 网络。

**典型用途：**
- DDoS 攻击
- 发送垃圾邮件
- 钓鱼攻击
- 信息窃取
- 分发其他恶意软件

> [!warning] 规模惊人
> 据估计约 **25% 的互联网个人电脑**曾是某个僵尸网络的一部分。

### 拒绝服务攻击（DoS / DDoS）

> [!note] 定义
> 目标不是窃取或修改数据，而是通过产生过量负载来**阻止合法用户使用服务**。

**常见方式：** 发起大量半连接（SYN flood），耗尽服务器连接资源

**DDoS（分布式）：** 从多个地点（通常是僵尸机器）同时攻击同一目标

> [!warning] 矛盾点
> 某些安全措施本身会引入 DoS 漏洞，如"登录失败后增加延迟"——攻击者可以利用这一点锁定所有合法用户账户。

**著名案例：**

| 年份 | 目标 | 规模 |
|------|------|------|
| 2012 | 美国各大银行 | 峰值 >60 Gbps |
| 2018 | GitHub | 1.35 Tbps |
| 2020 | AWS | 2.3 Tbps（持续 3 天） |

### Rootkit

> [!note] 定义
> 获取 OS 级别最高权限，并通过**隐藏自身存在**来维持访问权限的恶意软件。

**三大目标：**
1. 无限制地在目标系统上运行
2. 对安全软件、OS 和用户完全不可见
3. 执行恶意载荷（窃取信息、安装其他恶意软件）

**如何隐藏自身：**

Rootkit 监控所有发往 OS 的 API 查询，过滤掉与自身相关的结果：
- 拦截对自身所在目录的 `ls` 查询
- 拦截对 `/proc/` 目录的查询（进程列表）
- 拦截对 Rootkit 修改过的文件的 `read` 请求

**技术手段：**

| 手段 | 原理 |
|------|------|
| **API 钩挂（Hooking）** | 修改 OS API 地址，指向自身恶意代码；可作用于用户态或内核态 |
| **藏于磁盘未用空间** | 文件系统 API 无法看到这部分空间；修改设备驱动在加载时执行 Rootkit |
| **感染 MBR** | 在 OS 启动之前就加载自身，控制整个启动过程 |

## 总结
```
保护（Protection）= 执行访问策略的机制（内部视角）
安全（Security）  = 对抗外部威胁（外部视角）

威胁类型总览：
┌─────────────────────────────────┐
│      针对运行程序的威胁          │
│  木马 → 后门 → 逻辑炸弹         │
│  → 栈溢出 → 病毒                │
├─────────────────────────────────┤
│   针对系统和网络资源的威胁       │
│  端口扫描 → 蠕虫 → 僵尸网络     │
│  → DoS/DDoS → Rootkit           │
└─────────────────────────────────┘
```

> [!tip] 对系统程序员的启示
> - 永远做**边界检查**（`strncpy` 而非 `strcpy`）
> - 不要留后门，即使是调试用的
> - 遵循最小权限原则设计系统
> - 理解攻击原理，才能写出安全的代码
